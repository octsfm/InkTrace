# AGENTS.md · 通用开发规则

> 适用于所有项目。项目特定内容在项目根目录的 PROJECT.md 中定义（如存在）。

---

## 工作流说明

本项目采用以下工作流：

```
设计阶段：由人在外部完成，产出文档手动存入 docs/，封版后不再改动
实现阶段：Codex 读取 docs/ 下的封版文档，严格按文档实现代码
```

Codex 的职责只有一件事：**按文档实现，不自由发挥。**

---

## 一、文档结构识别

每次任务开始时，先扫描 `docs/` 目录，识别实际存在的文档结构。

目录命名可能有以下两种风格，均视为有效：

```
带编号：docs/01_requirements/、docs/02_architecture/、docs/03_design/、docs/04_plan/
不带编号：docs/requirements/、docs/architecture/、docs/design/、docs/plan/
```

以实际存在的目录为准，不假设结构，不创建不存在的目录。

---

## 二、实现阶段执行顺序（不可跳过）

```
1. 扫描 docs/ 目录，了解当前文档结构
2. 识别本次任务涉及的模块
3. 按优先级查找并阅读相关文档
4. 输出"实现前确认"
5. 确认范围无冲突，才开始写代码
6. 写完代码补充测试
7. 输出"实现完成报告"
```

---

## 三、文档查找优先级

```
需求文档      →  确认要做什么
架构文档      →  确认分层和依赖方向
详细设计文档  →  确认当前模块怎么实现
开发计划      →  确认当前阶段范围，哪些做哪些不做
```

不要只读设计文档。需求和架构必须一并阅读。

---

## 四、找不到文档时

**立即停止，输出：**

```
停止：缺少以下文档，无法开始实现。

缺少的文档及期望路径：
- [文档类型]：docs/[期望子目录]/[期望文件名]
- （按实际缺少的逐条列出）

请补充对应文档后继续。
```

不猜测，不按代码惯性继续，不开始写任何代码。

## 五、实现前确认（必须输出）

```
## 实现前确认

### 任务目标
-

### 读取的文档
- （文件路径 + 关键结论）

### 本次允许的修改范围
-

### 本次不允许修改的范围
-

### 关键设计约束
- （来自文档，不是自己推断）

### 发现的冲突或缺口
- 无 / 或列出并停止
```

---

## 六、实现范围规则

以下内容除非任务明确要求，否则禁止：

- 修改无关模块
- 顺手重构或优化
- 修改文档、配置文件、CI 文件、package lock
- 提前实现未来阶段的功能
- 任何 Git 操作（提交、push、PR）

---

## 七、架构边界规则

遵守 `docs/` 中架构文档定义的分层。通用底线：

- 上层只通过接口访问下层，不得调用内部实现
- 业务逻辑不得散落在 UI 层或 API 层
- 数据访问层不得被 UI 层绕过
- 外部依赖不得在业务核心层直接引用

项目有特定约束时，以 PROJECT.md 的定义为准。

---

## 八、测试规则

每次代码修改必须补充或更新测试，至少覆盖：

- 正常路径
- 错误路径
- 边界条件
- 不允许的调用路径

**测试通过 ≠ 符合设计。测试验证行为，文档定义意图。**

---

## 九、冲突处理

文档之间或文档与代码存在冲突时，优先级：

**需求文档 > 架构文档 > 详细设计文档 > 开发计划 > 当前代码**

无法判断时，停止并列出冲突点，等待确认，不自行选择。

```
-

### 是否可进入下一步

```

---

## 十一、项目特定规则

项目根目录存在 `PROJECT.md` 时，还须遵守其中定义的：

- 项目专属安全红线
- 项目专属架构约束
- 模块边界和阶段划分

PROJECT.md 优先级高于本文件，但不得放宽本文件的底线要求。
