# 资产盘点与迁移清单

## 1. 目标

本清单用于把当前仓库中的已有能力按“保留、迁移、弱化、废弃”四类进行盘点，避免后续重构时：

- 重复造轮子
- 不必要地推倒已有稳定资产
- 继续误把旧主链当成新主链

它不是代码审计报告，而是服务于新主干实施的迁移地图。

---

## 2. 分类规则

### 2.1 保留

底层稳定、设计方向正确、可直接作为新主干基础能力的资产。

### 2.2 迁移

能力本身有价值，但当前挂载位置、命名、边界或流程语义不适合作为新主链原样继承，需要迁入新模块。

### 2.3 弱化

保留兼容价值，但不应继续扩张，也不应再作为产品主路径。

### 2.4 废弃

设计方向已明确不再采用、只会干扰未来主链理解的历史实现。

---

## 3. 当前建议保留的资产

### 3.1 模型基础设施

建议保留：

- Kimi 客户端
- DeepSeek 客户端
- LLM 工厂的基础 provider 获取能力
- 职责路由的核心语义

保留原因：

- 这是双模型系统的基础零件
- 已经有较明确的职责边界

### 3.2 基础领域存储

建议保留：

- 小说、章节基础存储
- PlotArc 基础表结构与 repo
- 基础 DTO / repo 形态中稳定部分

保留原因：

- 数据资产已经存在
- 完全推翻数据库代价过高

### 3.3 部分服务能力

建议保留其“能力”而不是“原位置”：

- 章节分析能力
- 全书分析能力中的可复用 prompt/builders
- 写作生成能力
- 校验与修订能力的局部链条

---

## 4. 当前建议迁移的资产

### 4.1 V2WorkflowService

现状：

- 集中了承接大量业务编排
- 历史上承担过过多业务细节

迁移策略：

- 不再作为未来主干中心
- 只保留过渡期协调价值
- 其内部能力应拆到新的：
  - AuthorAgent
  - 导入与接手链
  - 写作工作流
  - 校验修订工作流

### 4.2 ChapterAIService

现状：

- 承载了大量具体分析/规划/写作桥接逻辑

迁移策略：

- 分解为更明确的任务能力提供者
- 保留 prompt 与 builder 资产
- 避免继续作为“万用 AI 服务”扩张

### 4.3 现有前端页面

现状：

- 页面本身功能多，但职责边界不稳定

迁移策略：

- 不原样继承页面结构
- 只复用局部组件、样式、已有状态管理片段和接口调用经验

### 4.4 现有任务与日志能力

现状：

- 已有部分任务状态与日志事件

迁移策略：

- 迁移到统一 trace 与任务模型下
- 保留事件命名中可复用部分
- 清理杂散命名与不完整事件

---

## 5. 当前建议弱化的资产

### 5.1 旧详情页主路径

弱化原因：

- 旧详情页承担了过多职责
- 不适合作为未来产品主入口继续扩张

处理方式：

- 进入过渡维护
- 新页面逐步替代其角色

### 5.2 历史兼容写作入口

弱化原因：

- 主要服务于旧调用链
- 不适合作为未来写作闭环主入口

处理方式：

- 保留兼容
- 不再继续扩能力

### 5.3 旧组织/整理兼容逻辑

弱化原因：

- 当前已经有过多兼容层
- 后续应让新导入与接手链成为主路径

---

## 6. 当前建议废弃或明确退场的资产

### 6.1 主备模型语义

结论：

- 明确废弃

原因：

- 与当前职责语义冲突
- 会持续误导系统设计

### 6.2 把 workflow 当成完整智能体的旧叙事

结论：

- 明确废弃

原因：

- 未来主干必须是 AuthorAgent + Workflow，而不是只靠 workflow 假装智能体

### 6.3 直接依赖整本正文大包分析的主路径

结论：

- 明确废弃

原因：

- 已经证明不稳定
- 与章节工件优先原则冲突

### 6.4 用低质量 fallback 伪装模型正式结果

结论：

- 明确废弃

原因：

- 直接损害作者信任
- 破坏结构层可信度

---

## 7. 后端迁移建议

### 7.1 适合优先迁移的能力

- 模型职责路由
- 章节分析工件生成
- Story Model 汇总逻辑
- PlotArc 基础对象管理
- 问题单与修订链的局部能力

### 7.2 应暂时留在旧链但不再扩张的能力

- 旧 V2 API 的兼容入口
- 为旧页面保底的轻量兼容层

### 7.3 后续应清退的部分

- 旧巨型 workflow 中仅服务历史页面的逻辑
- 不再被新主链消费的兼容 DTO

---

## 8. 前端迁移建议

### 8.1 可复用资产

- 已有基础 UI 组件
- 局部交互测试
- 任务状态显示经验
- 请求取消与分层加载的已有修补成果

### 8.2 不应原样继承的内容

- 旧详情页整体结构
- 把所有能力叠在单页的布局方式
- 用大量条件渲染修补职责边界的实现方式

### 8.3 新前端优先迁移顺序

1. 概览台
2. 任务台
3. 结构台
4. 章节台
5. 写作台

---

## 9. 数据迁移原则

### 9.1 保留现有核心资产

应尽量保留：

- 小说数据
- 章节正文
- 大纲
- 已确认版本

### 9.2 允许重建的结构层

以下对象允许在新主链中重建：

- Story Model
- PlotArc
- MemoryView
- 部分章节分析工件

### 9.3 需要迁移映射的对象

- 任务记录
- 版本引用
- 锁定状态

---

## 10. 迁移优先级

### 第一优先级

- 状态/版本/任务基础设施
- 新 AuthorAgent 主入口
- 新导入与接手链

### 第二优先级

- 新写作与改写闭环
- 新质量控制闭环
- 新 PlotArc 主链

### 第三优先级

- 前端页面全面切换
- 旧链清退
