# 导入与接手架构

## 1. 目标

导入与接手架构用于定义系统如何：

- 导入完整小说
- 导入未完成小说
- 导入大纲、草稿或大纲加部分章节
- 接手已有作品并建立 Story Model
- 在不重喂整本原文的前提下重建结构理解

这个模块是产品成立的关键，因为系统不仅要“从零创作”，还必须能“接手已有小说”。

## 2. 设计原则

1. 导入不是简单落库，而是建立可继续创作的故事模型
2. 接手未完成作品是一级能力，不是兼容模式
3. 优先逐章分析，再汇总全书，不默认直接整本拼接分析
4. 重新整理默认复用已有章节工件，不默认重喂整本原文
5. 导入后的目标不是“已入库”，而是“进入可写状态”

## 3. 输入模式

系统至少支持以下导入模式：

### 3.1 完整小说导入

输入：

- 完整正文
- 可选大纲

目标：

- 建立全书 Story Model
- 建立 PlotArc
- 支持结构分析与后续仿写/续写/改写

### 3.2 未完结小说导入

输入：

- 已有章节正文
- 可选大纲
- 可选人物设定与世界设定

目标：

- 判断当前推进位置
- 识别未完成目标
- 建立续写入口

### 3.3 只有大纲导入

输入：

- 大纲
- 可选题材与风格要求

目标：

- 建立初始 Story Model 草稿
- 允许进入规划链，但不直接冒充已完成分析的正式小说模型

### 3.4 大纲加部分章节导入

输入：

- 大纲
- 若干已写章节

目标：

- 结合大纲与已有正文建立阶段性 Story Model
- 支持继续写后续章节

### 3.5 零散草稿导入

输入：

- 无完整章节边界的草稿或片段

目标：

- 尽量标准化为章节候选单元
- 无法标准化时保持草稿态，不强行进入 `ready`

## 4. 导入的标准目标

导入完成的定义不是“文件已读完”，而是以下四层目标达成：

1. 内容资产落库
2. 章节单元建立
3. 章节工件建立
4. Story Model 与 PlotArc 建立

只有完成前 3 层，小说才有资格进入结构建模；只有通过第 4 层和可写校验，小说才能进入 `ready`。

## 5. 标准导入链

推荐标准链如下：

1. 文件读取
2. 编码与文本标准化
3. 章节切分
4. 原文保存
5. 初始索引建立
6. 逐章轻量分析
7. 全书结构汇总
8. PlotArc 抽取
9. Story Model 写回
10. Memory View 重建
11. 可写校验

### 5.1 文件读取

职责：

- 读取正文文件
- 读取大纲文件
- 读取外部附带设定

要求：

- 保留原始文件来源信息
- 记录导入批次
- 保留源数据哈希

### 5.2 文本标准化

职责：

- 清理异常编码
- 标准化换行与空白
- 处理明显分章标记

要求：

- 不改写语义
- 不丢失原文

### 5.3 章节切分

职责：

- 把文本标准化为章节单元

要求：

- 优先使用显式章节标记
- 无显式章节时允许 fallback 切分
- fallback 切分结果必须标记为低置信章节边界

### 5.4 初始索引建立

职责：

- 建立章节号、标题、字数、顺序、哈希
- 建立导入批次与来源映射

这一步不调用大模型。

### 5.5 逐章轻量分析

职责：

- 为每一章建立轻量工件

至少产出：

- 章节摘要
- 冲突
- 章节作用
- 必续点
- 续写记忆

要求：

- 以单章为主，不直接吃整本正文
- 允许批处理，但输入单位仍是章节级

### 5.6 全书结构汇总

职责：

- 基于章节工件和大纲构建全书结构

要求：

- 优先使用章节摘要与章节分析工件
- 不默认塞大量章节正文 preview
- Kimi 做的是结构理解，不是全文阅读器

### 5.7 PlotArc 抽取

职责：

- 基于 Story Model 与章节工件抽取剧情弧

要求：

- 抽取结果必须标注可信度
- 低可信度弧不能直接进入正式规划链

### 5.8 可写校验

职责：

- 判断当前小说是否进入 `ready`

最低条件建议：

- 已有章节单元
- 已有章节工件
- 已有生效 Story Model
- 已有生效 PlotArc 或明确可替代规划依据
- 当前没有阻断型低可信结构结果

## 6. 接手模式

接手模式分三类：

### 6.1 结构接手

目标：

- 只理解已有作品
- 不直接进入写作

适用：

- 已完结小说分析
- 参考作品分析

### 6.2 续写接手

目标：

- 在已有未完结小说基础上继续创作

适用：

- 已写一部分正文，需要继续写下去

### 6.3 协作接手

目标：

- 作者提供已有内容，系统以“辅助重整 + 协作续写”模式工作

适用：

- 作品质量不稳定
- 设定缺失
- 大纲未成形

## 7. 重整模式

系统必须正式区分三种重整模式：

### 7.1 `refresh_view`

定义：

- 不重跑大模型
- 只重建 Memory View 和展示聚合层

适用：

- 视图脏数据修复
- 前端展示异常

### 7.2 `rebuild_global`

定义：

- 复用已有章节工件
- 重建 Story Model、PlotArc、Memory View

适用：

- 结构需要重整
- 不希望重新分析所有章节原文

这是默认的“重新整理”模式。

### 7.3 `full_reanalyze`

定义：

- 重新逐章分析
- 再重建全书结构与 PlotArc

适用：

- 原文变化大
- 章节工件明显不可信
- 导入逻辑发生重大变化

## 8. 导入失败与恢复

### 8.1 失败分级

- 输入失败
  文件不可读、切章失败、数据缺失
- 分析失败
  Kimi 调用失败、章节工件缺失
- 建模失败
  Story Model 不可信
- 可写失败
  结构模型虽有，但不足以进入 `ready`

### 8.2 恢复原则

- 失败后回退到上一个稳定状态
- 不删除已成功落库的源数据
- 允许从已有章节工件恢复
- 不允许把低质量分析结果直接用于写作

## 9. 状态联动

建议的状态联动如下：

- 导入完成：`created -> imported`
- 启动结构分析：`imported -> analyzing`
- Story Model 构建成功：`analyzing -> modeled`
- 可写校验通过：`modeled -> ready`
- 分析失败且不可恢复：`analyzing -> blocked`

## 10. 当前实现约束

后续重构时必须遵守：

1. 默认重整不能再走“整本正文直喂 Kimi”
2. 全书分析必须优先基于章节工件
3. 重新整理默认使用 `rebuild_global`
4. `full_reanalyze` 必须是显式动作
5. 低可信结构结果不得自动进入写作链
