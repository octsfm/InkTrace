# 可观测性与运维架构

## 1. 目标

可观测性与运维架构用于定义 InkTrace 如何记录、追踪、调试、评估、恢复和运营整个作者智能体系统。

InkTrace 不是单次调用模型的轻量工具，而是一个持续运行的小说创作系统。  
如果没有完整的可观测体系，系统将无法回答以下关键问题：

- 这次分析为什么失败
- 这次续写为什么跑偏
- 为什么上个版本好，这个版本差
- 任务卡在什么步骤
- 哪个模型、哪个 prompt、哪份策略导致了结果变化

因此，可观测性不是附加能力，而是系统主干。

---

## 2. 设计原则

### 2.1 任务全链路可追踪

每一次导入、分析、规划、写作、改写、校验、修订任务，都必须能追踪完整链路，而不是只记录最终结果。

### 2.2 结果必须可回溯到输入与策略

任何重要结果都必须能回溯到：

- 输入上下文摘要
- 模型职责
- prompt 版本
- 策略版本
- 上下文装配版本
- 输出版本

### 2.3 调试信息分层

系统要同时支持：

- 生产环境的安全摘要日志
- 开发环境的深度调试信息

不能为了调试直接在生产中无限记录全文输入输出。

### 2.4 失败必须留下诊断资产

失败不是“返回错误字符串”就结束。  
失败必须留下足够的信息，支持：

- 复现
- 对比
- 重试
- 定位根因

---

## 3. 观测对象

系统至少需要观测以下五类对象：

### 3.1 请求链路

例如：

- 前端请求
- API 调用
- 用户动作触发的任务

### 3.2 任务链路

例如：

- 导入任务
- 分析任务
- 写作任务
- 改写任务
- 校验任务

### 3.3 智能体运行链路

例如：

- 智能体接收意图
- 智能体决策下一步
- 智能体调用工作流
- 智能体处理失败与降级

### 3.4 模型调用链路

例如：

- Kimi 分析调用
- DeepSeek 写作调用
- Prompt 构建
- 上下文装配

### 3.5 版本与结果链路

例如：

- 新旧 Story Model 对比
- 新旧 PlotArc 对比
- 新旧章节版本对比
- 候选稿与终稿对比

---

## 4. Trace 模型

系统应建立统一 trace 模型。

至少应包含以下标识：

- `request_id`
- `task_id`
- `workflow_run_id`
- `agent_run_id`
- `model_call_id`
- `novel_id`
- `chapter_id`
- `version_id`

### 4.1 链路嵌套关系

典型关系如下：

`request_id -> workflow_run_id -> task_id -> agent_run_id -> model_call_id`

并不是所有链路都有全部层级，但需要支持关联。

### 4.2 Trace 传播要求

任意子任务、模型调用、重试调用都必须继承或关联上层 trace 标识，避免日志碎片化。

---

## 5. 必须记录的事件

系统必须至少记录以下事件类型：

### 5.1 任务级事件

- 任务创建
- 任务开始
- 任务进度更新
- 任务重试
- 任务降级
- 任务成功
- 任务失败
- 任务取消
- 任务恢复

### 5.2 智能体级事件

- 智能体意图识别
- 智能体决策完成
- 智能体选择工作流
- 智能体阻断执行
- 智能体请求确认

### 5.3 模型调用级事件

- 模型调用开始
- 职责路由完成
- Prompt 构建完成
- 上下文装配完成
- 模型返回成功
- 模型返回失败
- 同职责降级
- 输出低可信

### 5.4 结构结果级事件

- Story Model 生成
- PlotArc 抽取
- PlotArc 更新
- 章节分析完成
- 一致性问题单生成
- 修订完成

---

## 6. 模型调用摘要记录

每次模型调用必须至少记录摘要信息，而不是只记录“调用成功/失败”。

应至少记录：

- `model_call_id`
- 模型提供方与模型名
- 职责类型
- 输入对象类型
- 输入摘要
- 输入规模信息
- prompt 版本
- 策略版本
- 上下文装配版本
- 输出摘要
- 可信度
- 耗时
- token 使用量（若可得）
- 是否重试
- 是否降级

### 6.1 输入摘要要求

输入摘要不必保留全文，但必须能让开发者理解：

- 本次调用吃了哪些对象
- 规模大概多大
- 是否包含大纲、章节摘要、最近章节、PlotArc、人物设定等

### 6.2 输出摘要要求

输出摘要至少要支持：

- 结构结果的快速理解
- 长输出的重点提取
- 失败结果的错误摘要

---

## 7. 结果比较能力

系统必须支持对关键结果进行版本比较。

至少应支持比较：

- Story Model v1 与 v2
- PlotArc 集合的差异
- 章节正文不同版本差异
- 问题单前后差异
- 策略变更前后结果差异

比较能力的目标不是做复杂 diff UI，而是支撑：

- 为什么结果变差
- 为什么这次分析更稳定
- 为什么这次改写更像 AI

---

## 8. 失败、重试与恢复记录

### 8.1 失败记录

失败必须记录：

- 失败阶段
- 错误类型
- 错误摘要
- 是否可重试
- 当时上下文规模
- 使用的策略快照

### 8.2 重试记录

每次重试必须记录：

- 第几次重试
- 为什么重试
- 是否改变上下文、策略或模型
- 重试结果

### 8.3 恢复记录

恢复任务时必须记录：

- 恢复来源状态
- 是否使用已有中间结果
- 恢复点
- 恢复后的结果

---

## 9. 调试模式与生产模式

### 9.1 开发调试模式

在开发或测试环境中，允许记录更详细的信息，例如：

- 更详细的 prompt 摘要
- 更详细的上下文对象清单
- 更细颗粒度的阶段事件

### 9.2 生产模式

在生产模式中，默认只保留：

- 摘要级日志
- 关键对象引用
- 必要的错误信息
- 可重现任务的关键快照

必须避免因为调试而无限记录敏感全文。

---

## 10. 评估与质量运营

可观测性不仅服务于故障定位，也服务于质量优化。

系统应逐步支持：

- 导入质量评估
- PlotArc 稳定性评估
- 续写一致性评估
- 改写质量评估
- AI 痕迹风险评估

这些评估结果应与：

- 模型版本
- 策略版本
- prompt 版本
- 上下文版本

关联，支持长期优化。

---

## 11. 运维面板需求

即使前期没有完整后台面板，系统设计上也必须预留运维视图需求。

至少需要能查看：

- 当前活跃任务
- 失败任务
- 卡住的任务
- 最近模型错误
- 最近降级事件
- 小说最近结构重建记录
- 各类结果版本比较入口

---

## 12. 与前端的边界

前端不应直接承担系统级调试逻辑，但应能展示面向作者和运营的必要状态，例如：

- 任务进度
- 当前失败原因
- 最近一次分析/重整时间
- 是否正在重试
- 是否存在阻断问题

而更底层的 trace、模型摘要和策略快照，适合放在系统调试界面或开发工具中。

---

## 13. 当前重构落地要求

在当前重构中，至少需要先落地：

1. 统一 trace 标识体系  
2. 任务事件日志清单  
3. 模型调用摘要日志  
4. 失败/重试/降级记录  
5. 关键结果版本对比入口  
6. 可用于复现问题的最小调试快照
