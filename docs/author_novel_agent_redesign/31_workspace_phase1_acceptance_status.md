# 31 Workspace 第一阶段验收状态

## 1. 文档用途

本文件用于记录 Workspace 第一阶段前端实现的当前验收状态。

目标不是重复设计稿，而是回答三个问题：

1. 哪些主线已经可以视为完成
2. 还剩哪些有意识保留的兼容兜底
3. 当前还有哪些非阻塞但值得继续跟踪的风险

## 2. 当前结论

截至当前实现，Workspace 第一阶段前端主线已接近验收态。

但如果严格对照 30 份 UI 重构设计规范中偏 Phase D / Phase E 的“微观交互”要求，当前实现还不能算彻底完成。

可以把现状理解为：

- 主路径已经完成收口
- 主要工作台已经形成统一状态语义
- 任务、结果回流、恢复入口、Copilot 与壳层对象模型已经打通
- 剩余问题主要收敛为少量有意识保留的兼容兜底，而不是明显功能缺口

需要额外强调：

- 上述判断成立于“工作区主线、壳层统一、任务主线、结果回流主线”层面
- 不等同于“所有高保真微观交互均已对齐正式 UI 规范”

## 3. 已完成主线

### 3.1 入口与页面主路径

- Dashboard / 小说列表已默认进入 `Workspace`
- `/novel/:id` 已成为默认工作区入口
- 旧 `/write` 与章节编辑路径仅保留兼容跳转
- 旧 `NovelWrite.vue` 与对应状态/测试资产已删除

### 3.2 工作区壳层统一

- `NovelWorkspace` 已成为主要壳层
- `Overview / Writing / Structure / Chapters / Tasks / Settings` 已进入同一工作区边界
- 侧栏、顶栏、命令面板、最近对象、Copilot 会话已开始共享统一对象语义

### 3.3 状态与快照统一

- `resourceSnapshot` 已用于资源与项目状态总览
- `taskCenterSnapshot` 已用于任务状态、恢复入口和壳层统计
- `currentContextSnapshot` 已用于章节上下文、问题数和对象诊断
- `sessionTasks / taskCenter / currentTask` 已开始收口成统一任务主线

### 3.4 Writing 结果回流主线

- 候选稿、改写结果、问题结果已进入正式对象语义
- `writing-result` 已支持：
  - 任务动作回流
  - 写作页结果区恢复
  - 最近对象恢复
  - 命令面板恢复
  - Copilot 当前对象
  - 顶栏对象动作

### 3.5 状态体验统一

- 主要工作台已接入统一的空状态、错误状态、恢复状态面板
- `Overview / Structure / Chapters / Tasks / Settings` 已可以直接给出恢复动作

## 4. 当前验收证据

当前前端已验证：

- `workspace + novel + stores + router` 回归 `104/104` 通过
- `npm run build` 通过

这说明当前主线更多是在做收尾与一致性增强，而不是修明显断链。

## 5. 有意识保留的兼容兜底

以下内容目前仍然保留，属于有意识的过渡保护，不建议在没有替代物之前直接删除。

### 5.1 任务快照空壳兜底

`NovelWorkspace` 仍保留对空 `taskCenterSnapshot` 的兼容判断。

目的：

- 支持旧测试桩
- 支持仍未完全接入统一快照的过渡态
- 避免壳层统计在局部场景突然归零

建议：

- 等所有调用方与测试都稳定依赖 `taskCenterSnapshot` 后再移除

### 5.2 结果展示的 structural fallback

`detemplatedDraft.display_fallback_to_structural` 仍保留。

这不是历史包袱，而是当前改写结果展示链路的防护字段。

建议：

- 只有当改写结果的数据契约足够稳定时再考虑移除

### 5.3 路由兼容跳转

旧写作路径和旧章节编辑路径仍保留兼容重定向。

这不是当前主入口，但对旧书签、旧跳转链接和历史外链仍有保护作用。

建议：

- 等确认没有外部依赖这些路径后再考虑删除

## 6. 当前非阻塞风险

### 6.1 Workspace 包体积偏大

当前构建产物中，`NovelWorkspace` 主脚本体积仍较大。

这不阻塞第一阶段验收，但属于后续值得继续跟踪的工程风险，尤其在：

- 首次加载
- 热更新速度
- 后续继续扩展工作区能力

### 6.2 少量壳层与视图仍存在双来源保护

虽然主线已经明显统一，但个别位置仍保留：

- 快照优先
- 局部 state 兜底

这类逻辑目前主要是防退化保护，而不是产品缺口。

## 7. 严格对照 Phase D / E 的当前落地状态

以下内容已经从“空壳或占位”推进到“第一版真实链路已落地”，但如果把验收标准继续上提到最高保真版本，它们仍然存在继续精修空间。

### 7.1 TipTap 微观交互已落地第一版

当前写作台已经具备：

- 已安装 `@tiptap/extension-bubble-menu`
- 正文划词气泡菜单已接入
- 选中文本后可执行：
  - 解释片段
  - 改写片段
  - 审查片段
- 上述动作会把选区上下文送入 Copilot 聊天草稿，形成真实可用链路

当前仍未达到的更高保真层：

- 尚未形成选区级后端改写/审查专用接口
- 尚未形成正文内部逐片段替换确认流

### 7.2 行内 CandidateBlock / Diff 已落地第一版

当前写作台已不再只有右侧 `DraftPreviewTabs`：

- 候选稿会在正文区渲染为行内候选块
- 改写结果会在正文区渲染为前后对照 Diff 块
- 并已接入：
  - 覆盖正文
  - 插入正文末尾
  - 保存草稿
  - 回到右侧结果区

当前仍未达到的更高保真层：

- Diff 仍是段落级对照，不是逐字符 patch 视图
- CandidateBlock 仍是正文区结果块，而不是真正嵌入编辑器文档树的自定义节点

### 7.3 Chapter Manager 看板拖拽已落地第一版

当前章节看板已具备真实拖拽流转：

- 卡片支持 HTML5 `dragstart / dragend`
- 列支持 `dragover / dragenter / dragleave / drop`
- 落下后会：
  - 本地同步章节状态
  - 调用章节更新接口
  - 成功提示
  - 失败回滚

当前仍未达到的更高保真层：

- 还未引入更复杂的排序、占位预览或跨列重排体验
- 还不是基于专门拖拽库的更完整看板系统

### 7.4 Copilot Inspire 已落地第一版独立推荐流

当前 `Inspire` 已不再只是占位说明和 `suggestedActions`：

- 已具备独立 `inspireItems` 数据模型
- 会按当前对象、结果类型、结构状态、失败任务生成本地灵感卡
- 会在切到 Inspire tab 时调用项目 `branches` 接口
- 后端返回的剧情分支会被转换成 Inspire 卡
- 每张卡支持：
  - 注入灵感
  - 立即发送
  - 打开对象

当前仍未达到的更高保真层：

- 还未接入更丰富的章节规划级后端推荐结果
- 还未形成长期记忆驱动的个性化推荐排序
- 还不是完整的多源推荐引擎

## 8. 当前判断

如果只看前端主线，Workspace 第一阶段已经非常接近“可以进入最终收口”的状态。

更准确地说：

- 现在最值得做的不是再大改主结构
- 而是继续收掉尾项、压缩兼容分支、确认哪些兜底可以在下一阶段移除
- 同时明确区分“第一版真实链路已落地”和“最高保真版本仍可继续精修”这两个层级

## 9. 下一步建议

建议后续按以下顺序继续推进：

1. 对兼容兜底做一次逐项清单化
2. 标记“必须保留 / 可下一阶段移除 / 可立即删除”
3. 对已落地的微观交互做第二阶段高保真打磨：
   - Bubble Menu 直连更细粒度 AI 动作
   - CandidateBlock / Diff 升级到更精细的编辑器内表达
   - 看板拖拽升级排序与占位反馈
   - Inspire 继续接入章节规划级后端增强
4. 对 `NovelWorkspace` 大包体积做一次拆分评估
5. 再做一次最终阶段验收回归，形成正式收口结论
