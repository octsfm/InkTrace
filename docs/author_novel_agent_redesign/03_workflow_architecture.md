# 工作流架构

## 1. 目标

工作流架构用于定义系统中的核心流程如何被组织、执行、暂停、恢复、取消、重试和完成。

工作流回答的是：

- 某件事要经过哪些步骤
- 每个步骤由谁做
- 何时进入下一步
- 失败后如何恢复

工作流本身不等于状态架构，也不等于智能体架构。

## 2. 设计原则

1. 所有长动作都必须转化为正式工作流
2. 工作流必须通过任务驱动，而不是页面直接串接口
3. 工作流步骤必须有清晰边界，不把多类职责塞进一个大函数
4. 工作流失败后必须能回退到稳定状态
5. 工作流执行必须可观测、可中断、可恢复

## 3. 工作流层级

系统建议分三层：

1. 用户级工作流
2. 系统级工作流
3. 模型级步骤

### 3.1 用户级工作流

面向作者理解的动作，例如：

- 导入小说
- 重新整理
- 继续写下一章
- 改写当前章节
- 修订问题稿

### 3.2 系统级工作流

系统内部为实现用户目标而编排的正式流程，例如：

- 逐章分析链
- 全书建模链
- PlotArc 抽取链
- 校验修订回环

### 3.3 模型级步骤

最小执行单元，例如：

- Kimi 章节分析
- Kimi 选弧
- DeepSeek 结构写作
- Kimi 一致性校验

模型级步骤不应直接暴露为用户级工作流。

## 4. 核心工作流目录

系统至少应有以下七类工作流：

1. 导入工作流
2. 分析工作流
3. 规划工作流
4. 写作工作流
5. 改写工作流
6. 校验与修订工作流
7. 恢复工作流

## 5. 导入工作流

### 5.1 目标

完成从原始输入到可写小说项目的初始化。

### 5.2 标准步骤

1. 读取输入
2. 标准化文本
3. 拆章
4. 保存源数据
5. 建立章节索引
6. 逐章分析
7. 汇总 Story Model
8. 抽取 PlotArc
9. 写回记忆
10. 可写校验

### 5.3 结束条件

- 成功：小说进入 `ready`
- 失败：回退到 `imported` 或进入 `blocked`

## 6. 分析工作流

### 6.1 目标

在不进入写作的前提下，建立或重建小说结构理解。

### 6.2 典型触发

- 导入后首次建模
- 重新整理
- 刷新记忆
- 结构可信度下降

### 6.3 子类型

- `refresh_view`
- `rebuild_global`
- `full_reanalyze`

## 7. 规划工作流

### 7.1 目标

决定当前下一章该推进什么、怎么推进。

### 7.2 标准步骤

1. 读取当前小说与章节状态
2. 选择目标弧
3. 判断规划模式
4. 生成章节任务与章节计划
5. 产出可写基线

### 7.3 关键规则

- 没有 `ready` 状态不得进入正式规划
- 没有可信 PlotArc 时必须走替代规划策略

## 8. 写作工作流

### 8.1 目标

产出可校验、可修订、可确认的正文草稿。

### 8.2 标准步骤

1. 装配续写上下文
2. DeepSeek 生成结构稿
3. DeepSeek 去模板化
4. 形成候选草稿
5. 进入校验链

### 8.3 关键规则

- 没有章节计划不直接生成正式正文
- 没有可用上下文不得进入写作

## 9. 改写工作流

### 9.1 目标

对已有正文进行重写、润色、风格调整或压缩扩写。

### 9.2 标准步骤

1. 理解改写目标
2. 装配改写上下文
3. 生成改写约束
4. DeepSeek 执行改写
5. 进入校验链

### 9.3 关键规则

- 改写默认生成新版本，不覆盖原稿
- 锁定内容不可被自动改写

## 10. 校验与修订工作流

### 10.1 目标

判断结果是否达标，并在必要时进入问题单修订回环。

### 10.2 标准步骤

1. Kimi 一致性校验
2. 生成问题单
3. 判断阻断级别
4. 需要修订时进入修订任务
5. 修订后复校
6. 通过后进入确认或发布

### 10.3 关键规则

- 阻断型问题不得直接提交为正式正文
- 问题单必须与任务、版本、模型调用可追溯绑定

## 11. 恢复工作流

### 11.1 目标

在任务中断、失败或进程重启后恢复系统运行。

### 11.2 标准步骤

1. 读取持久化任务状态
2. 判断是否存在可恢复点
3. 重新装配执行上下文
4. 从上一个稳定步骤恢复

### 11.3 关键规则

- 恢复必须基于持久化状态，不依赖前端本地状态
- 恢复必须显式记录为新的恢复事件

## 12. 工作流与任务关系

建议关系如下：

- 一个用户动作可触发一个主工作流
- 一个主工作流可拆成多个任务
- 一个任务可包含多个模型步骤

例如：

- `continue_novel_workflow`
  - `planning_task`
  - `writing_task`
  - `consistency_check_task`
  - `revision_task`

## 13. 工作流与状态关系

工作流不直接取代状态，但会驱动状态变化。

例如：

- 导入工作流成功驱动 `created -> imported`
- 分析工作流成功驱动 `imported -> modeled -> ready`
- 写作工作流开始驱动 `ready -> writing`
- 写作工作流成功驱动 `writing -> ready`

## 14. 暂停、恢复、取消、重试

### 14.1 暂停

仅允许发生在长任务的安全边界：

- 章节之间
- chunk 之间
- 模型调用完成后

### 14.2 恢复

恢复必须重新装配上下文，并基于持久化进度继续。

### 14.3 取消

取消后：

- 当前任务进入 `cancelled`
- 小说回退到上一个稳定状态
- 中间产物保留为历史，不直接清除

### 14.4 重试

重试必须记录：

- 原任务 ID
- 重试次数
- 是否使用降级策略

## 15. 当前重构要求

后续重构实现必须满足：

1. 详情页按钮不再直接等于工作流本身
2. “重新整理”必须区分具体模式
3. 写作、改写、修订都必须通过任务和工作流执行
4. 工作流服务必须做协调，而不是继续堆业务细节
