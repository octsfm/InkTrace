# InkTrace UI 工作区重构正式设计稿

## 1. 设计目标

InkTrace 的 UI 重构，不是为了“做得更好看”，而是为了让产品真正匹配它的核心定位：

一个以长篇小说创作为中心、由智能体参与协作、能够承接分析、续写、改写、结构管理与质量控制的小说创作工作系统。

UI 必须服务三个核心目标：

1. 让作者最快进入写作状态。
2. 让 AI 协作发生在写作过程里，而不是发生在页面外。
3. 让结构、任务、上下文、审查这些复杂能力成为写作的支撑，而不是新的负担。

因此，这次 UI 设计的总原则是：

**字是主角，编辑器是核心，UI 是容器，智能体是协作者。**

## 2. 整体信息架构

整个系统只保留两层：

### 2.1 Dashboard

负责管理“我有哪些小说、当前应该进入哪一本、最近在做什么”。

### 2.2 Novel Workspace

负责某一本小说的完整工作环境。进入之后，不再通过多个页面跳来跳去，而是在一个统一工作区中切换不同视图。

整体结构如下：

```text
Dashboard
  └── Novel Workspace
        ├── Writing
        ├── Overview
        ├── Structure
        ├── Chapter Manager
        ├── Tasks & Audit
        └── Settings
```

这里有一个非常重要的原则：

**不再保留独立的“项目管理页”和“小说详情跳板页”。**

外部入口只有 Dashboard。进入某本小说后，直接进入该小说的工作区。

## 3. 核心使用路径

UI 的主路径必须固定为：

打开系统  
-> 找到要继续写的小说  
-> 进入小说工作区  
-> 打开当前章节  
-> 写作 / AI 协作  
-> 审查 / 修订  
-> 保存 / 继续写

所有页面和交互，都要围绕这条路径判断是否合理。

如果一个设计不能帮助用户更快进入这条路径，它就不是核心能力，只能是辅助能力。

## 4. 视觉与布局总原则

### 4.1 总体气质

整体界面应呈现以下感觉：

安静、专业、克制、高留白、长时间使用不疲劳。

不应呈现为：

后台管理系统、功能集合页、营销式卡片界面、信息密度混乱的控制台。

### 4.2 基础配色

主背景建议使用纯白或极浅灰，例如接近 `#F8FAFC` 或 `#F9FAFB`。  
编辑器区应尽量保持更纯净的白色。  
侧边区域可以略微偏灰，用于建立区分，但不能使用大面积高饱和暖色或深紫色渐变。

主题色只用于：

- 主按钮
- 当前激活状态
- 进度线 / 少量强调
- 选中高亮

禁止把主题色大面积铺在 Header、卡片背景、区块底色上。

### 4.3 卡片使用原则

中央核心区原则上不使用卡片包裹。  
侧边辅助信息可使用轻卡片。  
卡片不依赖重阴影。  
尽量用留白、字体层级、细边框建立结构。

### 4.4 页面层级组织方式

视觉层级主要通过以下方式建立：

- 字号
- 字重
- 间距
- 对齐
- 细边框
- 适度底色差异

而不是依赖：

- 大块颜色
- 多重阴影
- 粗重卡片边框
- 大面积渐变

## 5. Dashboard 详细设计

### 5.1 页面定位

Dashboard 是整个产品的唯一总入口。  
它回答的问题不是“系统有哪些功能”，而是：

- 我有哪些小说
- 我现在应该先进入哪一本
- 最近发生了什么
- 我能不能快速开始继续写

因此它本质上是一个“工作入口页”，不是后台管理页，也不是书架展示页。

### 5.2 页面结构

页面分为三部分：

#### 顶部区域

顶部保持极简，只包含：

- 产品标识
- 全局搜索 / 命令面板入口
- 新建小说按钮
- 导入小说按钮
- 用户菜单

#### 主内容区域

主内容区域默认展示“我的小说”列表，以卡片流形式排列。

#### 辅助区域

可在上方或列表前提供极简的“最近继续创作”“最近任务”模块，但不应抢占视觉中心。

### 5.3 小说卡片设计

每张小说卡片必须是一个“工作入口”，不是一个展示模板。

卡片内建议包含：

- 小说标题
- 题材标签
- 当前状态，例如“创作中”“暂停中”“已完成”
- 当前进度，例如“第 142 章 / 32.7 万字”
- 最近编辑时间
- 主按钮：“继续写作”
- 次级入口：“概览”“结构”“任务”，可放在更多菜单里

### 5.4 排序逻辑

小说默认排序规则建议为：

1. 最近编辑优先
2. 有未完成任务的优先
3. 正在创作中的优先
4. 其他按更新时间排序

### 5.5 空状态设计

当用户还没有小说时，Dashboard 要明确引导：

- 创建新小说
- 导入已有小说
- 从大纲开始

## 6. Novel Workspace 总体设计

### 6.1 工作区定位

Workspace 是某一本小说的唯一工作环境。进入后，用户不应该感觉自己在几个页面之间跳，而应该感觉自己进入了一个“写作桌面”。

所有与这本小说相关的内容都在这里完成：

- 写作
- 查看概览
- 管理结构
- 管理章节
- 查看任务
- 使用 AI 协作

### 6.2 工作区总体布局

工作区采用固定四区结构：

- 最左：全局视图导航
- 左二：目录 / 树 / 结构导航
- 中间：核心内容区
- 右侧：AI Copilot 区

建议宽度关系如下：

- 最左导航栏：56px - 64px
- 左二栏：240px - 280px
- 中央内容区：自适应，占最大宽度
- 右侧 Copilot：300px - 360px

右侧栏和左二栏都应支持折叠。中央区域永远是主角。

### 6.3 顶部区域

Workspace 顶部只保留轻量面包屑与当前对象信息，例如：

`我的小说 / 缅北迷局 / 第 142 章`

右侧可放：

- 保存状态
- 当前字数
- 当前任务轻状态
- 快捷操作入口

### 6.4 底部状态栏

底部可以增加一条极简状态栏，用于承载：

- 当前字数
- 选中文字数
- 保存状态
- 任务进度
- 生成状态
- 快捷键提示

## 7. Workspace 四区详细设计

### 7.1 最左侧：主视图导航栏

这是工作区内的一级视图切换器，不是功能按钮堆。

建议只保留图标加短标签，包含：

- Writing
- Overview
- Structure
- Chapters
- Tasks
- Settings

设计要求：

- 宽度极窄
- 垂直排列
- 当前视图高亮
- 支持 tooltip
- 不能和章节树混在一起

### 7.2 左二栏：目录与对象导航区

这个区域根据主视图不同，内容会变化。

在 Writing 视图中显示章节目录树，包括：

- 卷
- 章
- 可选节
- 草稿 / 废稿

在 Structure 视图中显示结构对象导航，例如：

- 人物
- 世界观
- PlotArc
- 关系图
- 风险点

在 Tasks 视图中显示任务分类与筛选入口。

设计要求：

- 支持折叠 / 展开
- 支持搜索过滤
- 当前对象高亮
- 支持拖拽排序
- 与主导航明确区分层级

### 7.3 中央区域：核心内容区

中央区域必须根据当前主视图切换内容，但始终有一个统一原则：

**中央区域永远显示当前最重要的工作对象，不允许被大量辅助信息占满。**

#### Writing 视图

中央必须是编辑器。

不允许：

- 中间堆统计卡片
- 中间放大面积“建议下一步”
- 中间被仪表盘占据
- 中间成为说明书

#### Overview 视图

中央显示小说的概览信息，但仍然要保持简洁，聚焦：

- 当前创作状态
- 最近进度
- 最近章节
- 当前推荐动作
- 近期任务摘要

#### Structure 视图

中央显示结构内容，例如：

- Story Model 摘要
- 当前活跃弧
- 人物详情
- 世界观详情
- 风险分析结果

#### Chapters 视图

中央可显示章节列表、章节批量管理内容或章节看板。

#### Tasks 视图

中央显示当前任务、历史任务、失败任务、恢复入口。

### 7.4 右侧栏：Copilot 区

右侧栏不是“信息堆放处”，而是智能体协作区。

建议固定为三个 Tab：

- 对话
- 上下文
- 灵感

#### 对话 Tab

用于自然语言协作、提问、请求改写、请求补全。

#### 上下文 Tab

必须明确告诉用户当前 AI 使用的上下文，包括：

- 当前章节
- 当前目标弧
- 当前焦点角色
- 当前故事阶段
- 当前引用设定
- 当前任务快照

#### 灵感 Tab

显示轻量建议，例如：

- 三种下一章方案
- 三种改写方向
- 角色互动建议
- 冲突增强建议

灵感点击后默认不直接重型执行，而是先转为预览或候选动作。

## 8. Writing 视图详细设计

### 8.1 视图定位

Writing 是整个系统的默认主视图。  
用户进入某本小说后，默认应进入上次编辑位置；如果没有上次编辑位置，再退回最近章节；如果没有可编辑章节，再退回 Overview。

### 8.2 中央编辑器

必须使用 TipTap 或 ProseMirror 级别的编辑器体系，不能再使用原生 textarea。

编辑器应支持：

- 富文本基础排版
- 标题 / 段落 / 引用块
- 选区操作
- Slash 命令
- 行内 AI 建议
- 流式插入
- 锚点定位
- 轻量 diff 展示
- issue 装饰

### 8.3 编辑器状态体系

编辑器中的内容至少分为以下正式状态：

- 正式文本
- 候选文本
- AI 生成中临时文本
- 被 issue 标记的文本
- 刚接受修订的文本
- 被拒绝建议的区域

### 8.4 空状态

当未选择章节时，中央区域显示引导：

- 打开现有章节
- 创建新章节
- 从大纲生成章节

当章节为空白时，可显示轻量建议：

- 继续上一章
- 根据大纲开始
- 插入一段环境描写
- 生成下一段冲突

### 8.5 编辑器内 AI 交互

#### 划词菜单

选中一段后，应弹出操作条，提供：

- 润色
- 扩写
- 缩写
- 改语气
- 增强描写
- 转为对话
- 去 AI 味

#### Slash 命令

输入 `/` 时弹出命令菜单，例如：

- `/继续续写`
- `/根据大纲生成`
- `/补环境描写`
- `/增加冲突`
- `/补一句心理描写`

#### 候选块机制

AI 生成内容不应直接覆盖正式文本。默认以候选块形式出现，用户可：

- 接受
- 局部接受
- 编辑后接受
- 拒绝

### 8.6 保存与状态显示

Writing 视图应清晰显示：

- 自动保存中 / 已保存
- 当前任务运行状态
- 是否存在阻断 issue
- 当前版本状态（草稿 / 正式）

## 9. Overview 视图详细设计

Overview 是“小说工作区概览”，不是跳板页。

建议分为四块：

- 当前状态
- 最近进度
- 当前建议动作
- 近期任务

## 10. Structure 视图详细设计

Structure 用于服务“理解和规划”，不是日常写作默认入口。

左二栏可分为：

- Story Model
- PlotArc
- 人物
- 世界观
- 关系图
- 风险点

第一阶段关系图只做简化关系图。

## 11. Chapter Manager 视图详细设计

Chapter Manager 用于章节级管理，而不是正文写作本身。

包括：

- 章节目录树
- 章节状态
- 最近修改时间
- 批量操作
- 新建 / 移动 / 删除 / 排序

建议支持两种模式：

- 列表模式
- 看板模式

## 12. Tasks & Audit 视图详细设计

Tasks 不是给用户“看日志”，而是给用户“理解当前系统状态和恢复任务”。

建议分为：

- 当前运行任务
- 最近失败任务
- 最近完成任务
- 审查历史
- 恢复 / 重试入口

## 13. Command Palette 与全局搜索

这是 Workspace 和 Dashboard 之上的统一操作入口。

支持：

- `Cmd + K`
- `Ctrl + K`

可搜索：

- 章节
- 人物
- PlotArc
- 任务

可执行：

- 导出
- 进入视图
- 创建章节
- 发起审查

## 14. 导出与编译

导出不是写作主流程，但它是产品闭环的一部分。

建议入口放在：

- Overview 的小说级操作区
- Chapter Manager 的批量操作区

第一阶段建议支持：

- TXT
- DOCX
- EPUB

默认导出规则：

- 只导出正式文本
- 按章节顺序拼接
- 不导出候选态和临时态

## 15. 页面切换与导航规则

### 15.1 进入规则

- 从 Dashboard 点击小说，直接进入 Workspace
- 默认进入 Writing
- 默认打开上次编辑位置
- 没有上次编辑位置则打开最近章节
- 没有可编辑章节时退回 Overview

### 15.2 Workspace 内规则

- 一级视图切换只切视图，不离开 Workspace
- 点击章节默认进入或定位到 Writing
- 点击 issue 默认只定位正文
- 点击结构建议默认侧边预览，不强制切页
- 任务完成后不自动强行跳转

### 15.3 自动跳转规则

允许自动跳转的场景必须很少，仅限：

- 用户明确点击“查看正文”
- 用户明确点击“查看结构”
- 当前对象已不存在
- 权限或数据状态冲突

## 16. 单对象打开模型

正式可打开对象包括：

- 小说概览
- 章节
- 结构视图
- 任务
- diff 会话

规则：

- 同一章节默认只保留一个主编辑实例
- 可允许一个只读对照实例
- 同一个 diff 会话不重复打开多次
- 分屏时优先采用“主编辑 + 只读参考”

## 17. Copilot 交互优先级

Copilot 必须明确什么时候是“对话模式”，什么时候是“任务模式”，什么时候只展示只读信息。

规则：

- 对话模式：允许提问、解释、轻建议
- 任务模式：突出当前生成 / 改写 / 审查任务状态
- 只读模式：只展示上下文和结构信息

附加规则：

- 正在流式生成时，禁止继续发起新的重型写作任务
- Context 在任务启动后冻结为任务快照，任务结束后再刷新
- Inspire 点击后默认进入预览或候选动作，不直接重型执行

## 18. 编辑器内容状态定义

编辑器里的内容必须有正式状态定义：

- 正式文本
- 候选文本
- AI 正在生成的临时文本
- 已标记问题的文本
- 已接受修订
- 已拒绝修订

原则：

- 候选态不等于正式正文
- 临时生成态不等于已保存
- diff 未确认不改变正式版本
- 正式版本必须由明确确认动作产生

## 19. 任务与正文联动规则

- 写作生成完成：默认回流到编辑器，以候选块显示
- 改写完成：默认打开 diff 视图，而不是直接覆盖正文
- 审查完成：默认生成 issue 标记，并在正文建立锚点
- 失败任务：不允许静默失败，必须显示原因和恢复入口

## 20. 首屏最小信息集

### Dashboard 首屏

- 我的小说
- 最近继续创作
- 最近任务
- 创建 / 导入入口

### Writing Studio 首屏

- 当前章节标题
- 编辑器
- 右侧当前上下文
- 最近 issue 数量
- 当前任务状态

### Tasks 首屏

- 当前运行任务
- 最近失败任务
- 可恢复入口

## 21. 空状态与异常状态设计

必须覆盖：

- 没有 Story Model 时，Structure Studio 显示什么
- 没有章节时，Writing Studio 怎么引导
- 分析失败时，Tasks 页怎么提示
- Context 暂时不可用时，右栏显示什么
- diff 没生成成功时，如何回退

原则：

- 说明原因
- 给出下一步
- 绝不沉默空白

## 22. 移动与聚焦原则

任何会打断写作心流的动作，默认都不能把用户强制带离中央编辑区。

例如：

- issue 点击后只定位，不整页跳走
- 审查结果优先侧边展开，不顶掉正文
- AI 生成优先插到当前上下文，而不是跳新页面

## 23. 操作优先级冲突规则

多个操作冲突时，优先级如下：

1. 写作生成
2. 改写 / 修订
3. issue 定位与局部处理
4. 结构查看
5. 浏览性操作

规则：

- 同一时刻只允许一个主写作型任务占据中央控制权
- 流式生成期间禁止发起新的重型写作任务
- issue 定位允许发生，但不应中断当前主任务

## 24. 编辑器数据一致性规则

必须明确：

- 候选块不参与正式保存
- 流式生成中断后内容默认仍是临时态或候选态
- diff 未确认前不影响正式版本
- issue 修复前可继续编辑，但阻断 issue 会阻止正式提交

核心原则：

**UI 展示不等于数据已落库。**

## 25. 任务到 UI 生命周期绑定

任务在 UI 中的完整路径必须可见：

1. 任务创建：状态栏、任务页、Copilot 任务模式出现痕迹
2. 任务运行：显示阶段、进度、可取消状态
3. 任务完成：结果按任务类型回流正文 / diff / issue / 结构页
4. 任务失败：显示失败原因、重试入口、恢复建议

## 26. 跨工作区一致性规则

同一数据在不同工作区的同步规则必须明确。

例如：

- Writing 改了正文，Chapter Manager 与 Overview 要同步
- Structure 修改了人物或设定，Writing 的 Context / hover 信息要更新
- Tasks 修复了 issue，Writing 中对应标记要同步变化

## 27. 事件模型

建议核心事件包括：

- `ChapterUpdated`
- `TaskStarted`
- `TaskCompleted`
- `TaskFailed`
- `IssueResolved`
- `ContextChanged`
- `StructureUpdated`
- `DraftAccepted`
- `DraftRejected`
- `VersionCommitted`

这些事件用于：

- UI 刷新
- 跨工作区同步
- 日志追踪
- 埋点分析

## 28. 乐观更新 vs 最终一致性

统一规则如下：

- 编辑器内局部交互允许乐观更新
- 结构、分析、任务、版本结果以后端结果为准

## 29. 撤销 / 回滚边界

必须明确三层：

- 编辑器级：普通 `undo / redo`
- 任务级：整次改写 / 整次生成回退
- 版本级：章节历史版本恢复

## 30. 最小可运行 UI 闭环

第一阶段 UI 至少要支持以下完整路径：

1. 打开项目
2. 打开章节
3. 在 Writing Studio 里续写一段
4. 看到候选块
5. 接受
6. 触发审查
7. 修一个问题
8. 保存成功

## 31. 结构与文档管理补充

UI 还应明确支持：

- 多级拖拽式目录树
- 章节 / PlotArc 看板
- 资料库入口
- 废稿与归档区

## 32. 上下文与世界观交互补充

为了把底层 RAG 和记忆系统可见化，UI 应逐步支持：

- 正文内联 `@角色 / @地点 / @设定`
- hover 卡片
- 当前焦点角色 / 弧 / 设定的即时可见
- 简化版关系图

## 33. AI 审查与修订比对补充

AI 审查结果不能只放在列表里，应逐步具备：

- diff 对比视图
- issue 面板
- 正文锚点高亮
- 行内接受 / 拒绝

第一阶段先做到：

- issue 列表 + 正文定位
- diff 面板

## 34. 创作心流保护补充

为了避免产品重新滑回“机械工作流”，UI 应提供：

- Zen Mode
- 分屏参考
- 不打断编辑器的 AI 协作
- 最小必要状态反馈

## 35. 第一阶段必须落地的能力

第一阶段至少先落这 6 类：

1. Dashboard 与 Novel Workspace 两层结构
2. 工作区四区布局
3. Writing 默认主视图
4. TipTap / ProseMirror 编辑器底座
5. 候选块 / diff / issue 的第一版闭环
6. 左侧对象导航与右侧 Copilot 协作区

## 36. 设计结论

InkTrace 的 UI 最终不应该是“AI 功能页集合”，而应该是：

**一个以编辑器为中心、以左侧结构组织和右侧智能体协作为两翼、支持写作、审查、改写、结构管理、任务恢复的现代小说创作工作系统。**
