# 作者小说智能体产品蓝图

## 1. 文档定位

本文档定义 InkTrace 的最终产品目标、模型职责、核心能力、页面边界与重构方向。

从本文档开始，以下旧思路统一废止：

- `DeepSeek 主模型 + Kimi 备用模型` 的主备叙事
- 把多个功能堆叠在单一详情页中的产品形态
- 把“工作流服务”误认为“作者智能体”的描述
- 以环境变量作为 API Key 主配置入口的说明

本文档是当前仓库后续重构的统一基线。

## 2. 最终产品定义

InkTrace 的最终产品不是“AI 写作按钮集合”，而是一个作者小说智能体系统。

它必须能够：

- 从零创作小说
- 分析已有小说
- 接手未完成小说
- 自动续写小说
- 改写与润色小说
- 在作者协作下生成大纲、章节计划、正文与修订版本

一句话定义：

> 这是一个基于 `Kimi + DeepSeek` 双模型协同的作者智能体系统，能够理解、接手、创作、续写、改写并长期维护小说的一致性与质量。

## 3. 用户目标

系统必须同时支持两类使用方式：

### 3.1 自动创作模式

作者给出题材、设定、卖点、风格、大纲要求后，系统自动完成：

- 世界观设计
- 人物设计
- 故事结构设计
- 章节规划
- 正文生成
- 一致性校验与修订

### 3.2 协作创作模式

作者主导，智能体辅助。作者可以随时要求系统：

- 分析当前小说
- 生成或补写大纲
- 生成章节计划
- 根据要求续写下一章
- 改写已有正文
- 去 AI 味
- 调整风格、节奏、视角
- 诊断跑偏、设定冲突、人物失真

## 4. 大模型职责

产品固定使用两类模型：

- `Kimi`
- `DeepSeek`

模型职责必须严格固定，不再使用跨职责主备语义。

### 4.1 Kimi 负责

- 全书分析
- 章节分析
- 续写记忆提取
- PlotArc 抽取
- 弧状态更新
- 弧选择
- 章节规划
- 作者意图理解
- 一致性校验
- 改写约束生成

### 4.2 DeepSeek 负责

- 结构稿写作
- 章节续写
- 自动创作正文
- 去模板化改写
- 风格改写
- 润色精修
- 扩写与压缩
- 标题补写

### 4.3 模型关系

固定原则：

- `Kimi = 大脑`
- `DeepSeek = 笔`

一句话：

> Kimi 决定故事往哪走，DeepSeek 把它写出来。

## 5. 核心能力矩阵

### 5.1 创作

输入：

- 题材
- 卖点
- 设定
- 风格要求
- 大纲要求

输出：

- 世界观
- 人物设定
- 大纲
- 章节计划
- 正文

### 5.2 分析

支持导入：

- 完整已完结小说
- 未完结长篇小说
- 部分章节
- 零散草稿
- 只有大纲
- 大纲 + 部分正文

分析输出：

- 人物图谱
- 世界设定
- 主线/支线
- PlotArc
- 当前进度
- 风格画像
- 风险点

### 5.3 接手

对未完成小说，系统必须能够：

- 自动理解已有章节
- 识别当前推进位置
- 补足结构认知
- 判断当前可续写方向
- 在原风格与原逻辑上稳定续写

### 5.4 续写

系统必须支持：

- 自动续写
- 按要求续写
- 按目标弧续写
- 按章节计划续写

### 5.5 改写

系统必须支持：

- 去 AI 味
- 去模板化
- 风格改写
- 节奏调整
- 视角改写
- 保持剧情不变的表达重写
- 扩写/压缩
- 润色精修

## 6. Story Model

所有导入、分析、创作、续写、改写都必须建立在统一的 `Story Model` 上。

`Story Model` 至少包含：

- 人物图谱
- 世界观与设定规则
- 主线与支线
- PlotArc 体系
- 当前推进状态
- 未完成目标与悬念
- 风格画像
- 一致性风险点

系统不应直接把“原文大段拼接”当成长期理解方式，而应先构建结构化故事模型。

## 7. PlotArc 定义

PlotArc 是正式中层叙事推进单位，不是章节摘要。

### 7.1 类型

- `main_arc`
- `character_arc`
- `supporting_arc`

### 7.2 状态

- `active`
- `dormant`
- `completed`
- `archived`

### 7.3 优先级

- `core`
- `major`
- `minor`

### 7.4 展示与理解视角

最终必须至少支持三层视角：

- 全书总弧
- 开局到当前阶段弧
- 最近几章局部推进弧

### 7.5 产品作用

PlotArc 用于：

- 全书结构理解
- 当前推进判断
- 下一章规划
- 长线一致性控制
- 作者解释与建议

## 8. 统一工作闭环

所有核心能力最终都应运行在同一闭环上：

`读取/导入 -> 理解 -> 建模 -> 选目标 -> 规划 -> 写作/改写 -> 校验 -> 修订 -> 写回记忆`

### 8.1 导入/分析链

推荐标准链：

1. 读文件
2. 拆章节
3. 保存章节原文
4. 逐章分析
5. 汇总全书分析
6. 抽取 PlotArc
7. 写回 Story Model
8. 构建轻量 Memory View

### 8.2 续写链

推荐标准链：

1. Kimi 选弧
2. Kimi 规划章节
3. DeepSeek 生成结构稿
4. DeepSeek 去模板化/风格改写
5. Kimi 一致性校验
6. DeepSeek 按问题单修订
7. 写回章节与记忆

### 8.3 改写链

推荐标准链：

1. Kimi 理解改写目标
2. Kimi 生成改写约束
3. DeepSeek 执行改写
4. Kimi 校验剧情保持与风格一致性
5. 必要时 DeepSeek 再修订

## 9. 产品界面边界

产品不应继续围绕“功能按钮”组织，而应围绕“作者与智能体协作”组织。

### 9.1 概览台

只负责：

- 小说基础信息
- 当前状态
- 最新整理结果
- 最近章节
- 下一步入口

不负责承载全部结构、全部章节、全部工作流控制。

### 9.2 结构台

负责：

- 全书结构
- PlotArc
- 人物
- 世界观
- 全书理解摘要
- 作者助手建议

### 9.3 写作台

负责：

- 续写
- 自动创作
- 改写
- 润色
- 章节计划协作

### 9.4 章节台

负责：

- 章节列表
- 章节导入
- 章节编辑
- 章节分析结果

### 9.5 任务台

负责：

- 整理任务
- 分析任务
- 校验任务
- 修订任务
- 错误与问题单

## 10. 智能体定义

当前仓库主链仍以 workflow 为主，尚未形成真正的作者智能体。

最终必须引入统一入口，例如：

- `AuthorAgentService`

它负责统一调度：

- 小说理解
- 目标判断
- 剧情弧选择
- 章节规划
- 写作调用
- 改写调用
- 校验与修订
- 记忆写回

它的职责不是直接生成所有文本，而是“组织整个创作闭环”。

## 11. 质量目标

系统产出的小说必须满足：

- 内容合理
- 不飞设定
- 人物不崩
- 长线不断
- 风格稳定
- AI 味少
- 模板化低

因此系统必须长期具备以下约束：

- 结构约束
- 设定约束
- 风格约束
- 去模板化约束
- 一致性校验
- 修订单闭环

## 12. 当前重构原则

后续开发必须遵循以下原则：

1. 先收口主链，再修零散细节 bug
2. 先验证核心创作闭环，再扩展功能
3. 先拆清页面职责，再补交互
4. 所有分析职责固定给 Kimi
5. 所有正文与改写职责固定给 DeepSeek
6. 不再回到跨职责主备 fallback 语义

## 13. 当前与目标的差距

截至当前仓库状态，主要差距包括：

- PlotArc 尚未完成产品级可靠性验证
- 页面功能堆叠严重，尤其是详情页职责过重
- 真正的 `AuthorAgentService` 尚未落地
- 现有 workflow 仍强于 agent
- 结构理解、续写、改写虽已接通，但尚未完全形成统一产品体验

## 14. 后续重构优先级

建议按以下顺序推进：

1. 收口最终产品定义与架构边界
2. 拆分页面职责，瘦身详情页
3. 建立最小可信主链：
   - 导入
   - 结构理解
   - PlotArc
   - 规划
   - 续写
   - 校验
   - 修订
4. 对 PlotArc 做完整验证
5. 落地统一 `AuthorAgentService`
6. 最后再系统性处理零散细节 bug

## 15. 最终一句话

InkTrace 的最终形态不是“多个 AI 功能页”，而是：

> 一个基于 Kimi 与 DeepSeek 双模型协同、能够理解已有小说、接手未完成小说、从零创作新小说，并与作者长期协作完成分析、续写、改写与创作的作者小说智能体。

## 16. 系统运行与生命周期模型

InkTrace 不只是一个具备创作能力的智能体系统，还必须是一个具备完整状态管理、任务调度、错误恢复与可观测能力的长期运行系统。

### 16.1 小说生命周期（Novel Lifecycle）

系统必须明确每一本小说在系统中的状态流转。

状态定义：

- `created`
  已创建但未导入内容
- `imported`
  已导入原文 / 大纲 / 草稿
- `analyzing`
  正在进行结构分析
- `modeled`
  已完成 Story Model 构建
- `ready`
  已具备续写 / 改写能力
- `writing`
  正在生成新章节
- `revising`
  正在进行改写或修订
- `completed`
  小说已完成

状态流转原则：

- `imported -> analyzing -> modeled -> ready` 为标准初始化路径
- 任意阶段失败必须可回退到上一个稳定状态
- `ready` 是所有写作行为的入口状态
- 写作、改写不会破坏已有结构模型，只会更新其状态

### 16.2 任务系统（Task System）

系统所有行为必须通过任务驱动，而不是直接调用模型。

任务类型：

- `analysis_task`：结构分析
- `planning_task`：章节规划
- `writing_task`：章节生成
- `rewrite_task`：改写
- `consistency_check_task`：一致性校验
- `revision_task`：修订

每个任务必须具备：

- `task_id`
- `novel_id`
- `task_type`
- `status`：`pending / running / success / failed`
- `progress`
- `retry_count`
- `created_at`
- `updated_at`

执行原则：

- 所有任务必须可中断、可恢复
- 长任务必须支持进度更新，至少支持章节级或 chunk 级更新
- 失败任务必须支持重试或降级执行
- 任务执行必须记录完整日志用于回溯

### 16.3 错误处理与降级策略

系统必须定义统一错误处理策略，而不是依赖单次模型成功。

错误类型：

- `timeout`
- `llm_error`
- `incomplete_result`
- `low_confidence`

处理策略：

- 超时：自动拆分任务，退化为章节级或 chunk 级执行
- 单次失败：自动重试，次数必须有限
- 多次失败：降级为简化分析
- 分析结果不可信：标记并禁止进入写作链

关键原则：

- 系统不得把低质量分析结果直接用于写作

### 16.4 用户控制权边界

系统必须明确作者与智能体之间的控制边界。

作者可控制内容：

- 指定写作方向（`plot_direction`）
- 指定或修改 PlotArc
- 修改人物设定与世界观
- 指定风格与节奏
- 选择是否采用某次生成结果

作者可锁定内容：

- 人物设定不可修改
- 世界规则不可变更
- 已完成章节不可覆盖

智能体职责边界：

- 不得主动修改已锁定内容
- 不得破坏已有结构一致性
- 所有改动必须可追溯

### 16.5 任务与生命周期联动

任务执行必须驱动小说状态变化。

示例：

- `analysis_task success`：`imported -> modeled`
- `writing_task start`：`ready -> writing`
- `writing_task success`：回到 `ready`
- `rewrite_task`：不改变主状态，仅更新内容版本

### 16.6 可观测性与调试能力

系统必须具备可观测能力，否则无法优化分析质量。

必须记录：

- 每次模型调用的输入与输出摘要
- 每个任务的执行链路（trace）
- 分析结果版本，且版本间可对比
- 错误与重试记录

目标：

- 支持分析质量评估
- 支持问题复现
- 支持结构对比
- 支持持续优化

## 17. 运行系统的最终约束

InkTrace 不仅是一个具备创作能力的智能体系统，更是一个具备完整状态管理、任务调度、错误恢复与可观测能力的长期运行系统。

系统必须保证：

- 可持续分析
- 可稳定续写
- 可回溯与可修复

否则无法支撑长篇小说创作场景。
