# 智能体架构

## 1. 目标

智能体架构用于定义真正的作者智能体 `AuthorAgent` 应该是什么、负责什么、不负责什么，以及它如何与工作流、任务、记忆、模型和用户交互。

当前系统的主链更接近 workflow orchestration，而不是完整作者智能体。  
这份文档的作用，就是把“智能体”从口号落到正式架构层。

## 2. 智能体的正式定义

`AuthorAgent` 不是单次请求封装，也不是多个按钮背后的服务拼装。

它是一个持续工作的决策层，负责：

- 理解小说当前状态
- 理解作者当前意图
- 决定接下来该做什么
- 调度分析、规划、写作、改写、校验、修订工作流
- 维护长期创作一致性

一句话：

> AuthorAgent 负责组织创作闭环，而不是亲自完成所有文本生成。

## 3. 智能体职责边界

### 3.1 智能体负责

- 目标判断
- 意图理解
- 小说当前状态判断
- 工作流选择
- 任务下发
- 任务结果汇总
- 风险判断
- 是否继续、暂停、阻断的决策

### 3.2 智能体不负责

- 直接保存源数据
- 直接管理数据库事务
- 直接取代章节编辑器
- 直接替代模型客户端
- 越过权限边界修改已锁定内容

### 3.3 模型职责仍然保留

- Kimi：理解、分析、规划、校验
- DeepSeek：写作、续写、改写

智能体本身不是第三个写作模型，而是调度与决策层。

## 4. 智能体入口

后续建议有一个统一入口，例如：

- `AuthorAgentService`

它应至少暴露四类入口能力：

1. `analyze_novel`
2. `continue_novel`
3. `rewrite_content`
4. `assist_author`

### 4.1 analyze_novel

用于：

- 导入后结构理解
- 完整小说分析
- 结构重整

### 4.2 continue_novel

用于：

- 自动续写
- 按要求续写
- 按目标弧续写

### 4.3 rewrite_content

用于：

- 改写章节
- 去 AI 味
- 风格调整
- 精修润色

### 4.4 assist_author

用于：

- 回答创作问题
- 提建议
- 给出多个方案
- 生成章节计划或大纲

## 5. 智能体的输入

智能体每次运行至少需要以下输入：

- `novel_id`
- 当前小说状态
- 当前用户意图
- 当前目标对象
- 当前锁定规则
- 当前生效 Story Model / PlotArc / 版本信息

用户意图必须结构化，不应只以自由文本直接驱动执行。

## 6. 智能体的输出

智能体输出的不是单一文本，而是以下三类结果之一：

1. 决策结果
2. 工作流触发结果
3. 解释结果

### 6.1 决策结果

例如：

- 当前应先重整，不应续写
- 当前应先修订上章
- 当前可推进主线弧

### 6.2 工作流触发结果

例如：

- 启动分析工作流
- 启动章节规划任务
- 启动写作任务

### 6.3 解释结果

例如：

- 为什么推荐这条弧
- 为什么阻断写作
- 为什么要求重新分析

## 7. 智能体运行阶段

一个标准的 Agent Run 至少分为：

1. 装配上下文
2. 理解意图
3. 判断当前小说可执行条件
4. 选择工作流
5. 下发任务
6. 等待与汇总
7. 输出结果

### 7.1 装配上下文

调用对应的上下文装配器，读取：

- 小说当前状态
- 生效版本
- 当前约束
- 当前任务目标

### 7.2 理解意图

判断作者此刻是在要求：

- 分析
- 续写
- 改写
- 解释
- 规划

### 7.3 判断可执行条件

例如：

- 小说是否为 `ready`
- 当前是否存在阻断型问题
- 当前是否缺失关键 Story Model
- 当前章节是否允许改写

### 7.4 选择工作流

智能体不直接跳到模型调用，而是先决定：

- 该走哪类工作流
- 需要哪些任务
- 是否允许降级

### 7.5 下发任务

由智能体触发任务与工作流，而非页面直接串调模型。

### 7.6 等待与汇总

智能体等待任务结果，汇总：

- 成功结果
- 问题单
- 失败原因
- 恢复建议

### 7.7 输出结果

最终给用户或上层系统：

- 可执行结果
- 解释说明
- 下一步建议

## 8. 智能体与工作流关系

建议关系是：

- 智能体负责判断和调度
- 工作流负责执行
- 任务负责承载执行状态
- 模型负责具体分析/写作能力

用一句话概括：

`Agent decides, workflows execute, tasks persist, models produce.`

## 9. 智能体与记忆关系

智能体不直接承担长期记忆存储，但必须能够使用和更新记忆。

### 9.1 智能体读取

- Story Memory
- Chapter Memory
- Continuation Memory
- 当前版本与锁定规则

### 9.2 智能体写回

智能体本身不直接写 raw memory 内容，而是通过工作流和写回服务完成：

- 分析结果写回
- PlotArc 写回
- 校验结果写回
- 问题单写回

## 10. 智能体与权限边界

智能体必须服从以下规则：

- 不得自动修改已锁定人物设定
- 不得自动覆盖已确认章节
- 不得绕过发布和确认机制
- 不得在低可信分析结果上直接启动写作

## 11. 智能体自治程度

建议定义三档自治模式：

### 11.1 `assist_only`

- 只给建议
- 不自动启动工作流

### 11.2 `guided_execute`

- 给建议
- 在用户确认后启动工作流

### 11.3 `auto_run`

- 在规则允许范围内自动推进工作流

默认建议使用：

- 协作创作：`guided_execute`
- 自动创作：`auto_run`

## 12. 智能体失败与恢复

### 12.1 失败类型

- 上下文不足
- 关键依赖缺失
- 下游任务失败
- 低可信结构结果

### 12.2 恢复原则

- 智能体失败不等于小说失败
- 智能体失败后应输出恢复建议
- 需要恢复时优先恢复任务链，而不是让用户手动猜下一步

## 13. 前端体现

前端最终不应暴露为“很多 AI 功能按钮”，而应体现为：

- 当前状态
- 当前建议
- 当前目标
- 当前任务
- 当前风险
- 下一步可选动作

也就是说，前端应表现为“与智能体协作”，而不是“自己拼工作流”。

## 14. 当前重构要求

后续重构实现必须满足：

1. 建立统一智能体入口
2. 页面不再直接等价于模型调用
3. 智能体必须以状态、版本、上下文为前提做判断
4. 智能体不能越过权限边界
5. 智能体输出必须包含解释能力，而不只是任务触发
