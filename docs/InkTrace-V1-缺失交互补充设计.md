# InkTrace V1 缺失交互补充设计（修正版：Web 导入）

更新时间：2026-04-29  
适用范围：V1（纯文本写作工具）  
目标：补齐 V1 当前实现中缺失的关键交互与入口，指导开发落地（不包含导入大纲）。

---

## 0. 背景与约束

### 0.1 背景

V1 主链路（书架 → 写作页、自动保存、离线回放、409 冲突）可用的前提下，当前仍存在影响“可用性/可发现性”的缺口：

1. 新建作品后直接进入写作页，但缺少“作品标题/作者”的输入入口（用户无法按自己的作品命名）。
2. 书架页缺少“删除作品”的入口（后端已支持删除但前端无触发点）。
3. TXT 导入在 Web 端应为“选择文件并上传”，而不是“输入本地路径让后端读取”。

### 0.2 硬约束（保持 V1 边界）

- 不新增“大纲导入/大纲管理/结构整理/AI 写作”能力。
- 不改变 V1 主链路：新建/导入成功后进入写作页 `/works/:id`。
- 保持 Local‑First 保存链路不受影响。
- **V1 形态为 Web**：导入必须支持浏览器端文件选择与上传。

## 1. 新建作品：标题/作者入口（Create Work Modal）

### 1.1 目标

- 用户在“新建作品”时可填写：
  - **作品标题**（必填）
  - **作者**（可选）
- 创建成功后仍然**直接进入写作页**（减少步骤）。

### 1.2 交互流程

1. 用户在书架页点击“新建作品”
2. 弹出 `CreateWorkModal`
3. 用户输入 `title` / `author`
4. 点击“创建并进入写作”
5. 调用 `POST /api/v1/works`
6. 成功后跳转 `/works/:id`

### 1.3 UI 规范

- 字段
  - 作品标题（必填）
    - 默认值：可沿用“未命名作品 + 日期”作为预填
    - 校验：空字符串禁止提交，提示“请输入作品标题”
  - 作者（可选）
- 按钮
  - 取消
  - 创建并进入写作（主按钮）
- 状态
  - 提交中：按钮禁用 + 文案“创建中…”
  - 失败：弹轻提示（Toast），不阻断页面

### 1.4 API 契约

- Request：`POST /api/v1/works`
  - body：`{ "title": string, "author": string }`
- Response：`{ id, title, author, current_word_count, created_at, updated_at }`

### 1.5 验收标准（DoD）

- 点击新建 → 必定出现弹窗
- 标题为空不能提交
- 创建成功后进入 `/works/:id`，书架卡片显示正确标题/作者

## 2. 书架作品卡：更多菜单与删除入口（WorkCard Actions）

### 2.1 目标

- 书架页每个作品卡片必须有可发现的 `(...)` 更多操作入口
- V1 最小闭环必须实现：**删除作品（带二次确认）**

可选（不作为本补丁强制项）：
- 导出 TXT
- 重命名/修改作者（需要后端新增 `PUT/PATCH /api/v1/works/{id}`，暂不强制）

### 2.2 交互流程（删除）

1. 用户点击作品卡片 `(...)`
2. 弹出菜单项：删除作品（危险色）
3. 点击删除后弹出二次确认 Dialog：
   - 文案：“此操作不可恢复，确认删除？”
   - 按钮：取消 / 删除
4. 点击“删除”：
   - 调用 `DELETE /api/v1/works/:id`
   - 成功后刷新作品列表（或从列表中移除该项）

### 2.3 UI 规范

- `(...)` 点击必须 `stopPropagation`，避免误触进入写作页
- 删除使用危险样式（红色）
- 二次确认必须存在

### 2.4 API 契约

- Request：`DELETE /api/v1/works/{work_id}`
- Response：`{ ok: true, id: work_id }`

### 2.5 验收标准（DoD）

- 作品卡片点击主体仍进入写作页
- 点击 `(...)` 不会跳转
- 删除需要二次确认
- 删除成功后列表立刻更新，刷新页面作品不存在

## 3. TXT 导入（Web）：选择文件并上传（替代 file_path 导入）

### 3.1 结论（必须写死）

在 Web 端，“输入本地路径让后端读取文件（file_path）”不可作为产品方案。  
V1 的 TXT 导入必须改为：**选择文件 → 上传文件内容 → 后端解析入库**。

### 3.2 交互流程

1. 用户点击“导入 TXT”
2. 弹窗内点击“选择文件”
3. 选择 `.txt` 文件后，UI 展示文件名
4. 可选填写：作品标题、作者
5. 点击“开始导入”
6. 前端以 `multipart/form-data` 上传文件
7. 后端解析分章、创建作品、返回 work DTO
8. 前端刷新书架列表并进入写作页 `/works/:id`

### 3.3 UI 规范

- “TXT 文件”输入框为只读展示（文件名），不允许手输路径
- 未选择文件点击导入：提示“请先选择 TXT 文件”
- 导入中：按钮禁用 + “导入中…”
- 成功：Toast “导入成功” 并跳转

### 3.4 API 契约（新增接口）

- Request：`POST /api/v1/io/import-upload`
  - `Content-Type: multipart/form-data`
  - form：
    - `txt_file`: File（必填）
    - `title`: string（可选）
    - `author`: string（可选）
- Response：work DTO：`{ id, title, author, current_word_count, created_at, updated_at }`

### 3.5 兼容与灰度（可选，但推荐）

如果当前已经存在 `POST /api/v1/io/import`（file_path 模式）用于本地开发调试，建议：

- 默认在生产禁用该路径导入
- 仅当 `INKTRACE_ALLOW_PATH_IMPORT=true` 时才允许调用
- 否则返回 `400 path_import_not_supported_in_web`

避免 Web 用户误用导致“看起来像 bug”。

### 3.6 验收标准（DoD）

- Web 端点击导入必定打开系统文件选择器
- 选择 txt 后导入成功，作品出现在书架
- 进入写作页后章节列表可见（解析正确或按兜底规则生成单章）
- 乱码处理：至少保证不崩溃；无法解码时给出明确错误提示

---

## 4. 不在本补丁范围内

- 导入大纲 / 大纲管理 / 结构整理 / AI 写作入口
- Web 上传进度条与断点续传（可后续增强）
- 作品重命名 / 修改作者（除非后端补 `PUT/PATCH /api/v1/works/{id}`）
