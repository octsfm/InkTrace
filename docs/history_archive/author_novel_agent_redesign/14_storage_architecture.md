# 存储架构

## 1. 目标

存储架构用于定义哪些数据必须持久化、哪些数据可缓存、哪些数据可重建、哪些运行态必须跨进程保留。

它直接回答：

- 什么进数据库
- 什么进文件存储
- 什么只做缓存
- 什么必须长期保留
- 什么可以丢掉后重建

这个系统的数据类型很多，至少包括：

- 原文
- 大纲
- 章节
- Story Model
- PlotArc
- 记忆
- 任务
- 日志
- 问题单
- 模型调用记录

## 2. 设计原则

1. 源数据与派生数据必须分离
2. 领域持久化与运行态持久化必须分离
3. 可重建数据不应承担唯一事实来源职责
4. 进度、任务、错误等运行关键数据不得只留在内存
5. 存储设计必须服务版本架构与状态架构

## 3. 数据分类

系统数据分成六类：

1. 源数据
2. 领域事实数据
3. 派生结构数据
4. 运行态数据
5. 可重建视图数据
6. 日志与追踪数据

## 4. 源数据

### 4.1 定义

源数据是导入后不可替代的原始资产。

包括：

- 原始小说正文
- 原始章节正文
- 原始大纲
- 用户手工输入的关键设定
- 用户上传的参考资料路径或标识

### 4.2 规则

- 必须持久化
- 默认不可覆盖
- 只能生成新版本，不能直接丢失

### 4.3 建议存储

- 结构化元数据进数据库
- 长文本正文可进数据库或文件存储，但必须可稳定回溯

## 5. 领域事实数据

### 5.1 定义

领域事实数据是系统后续所有行为的正式依据。

包括：

- 小说基本信息
- 章节实体
- 当前小说状态
- 当前章节状态
- 生效中的 Story Model 版本标识
- 生效中的 PlotArc 版本标识
- 锁定规则
- 发布信息

### 5.2 规则

- 必须持久化到数据库
- 必须具备事务一致性
- 不允许只存在缓存

## 6. 派生结构数据

### 6.1 定义

派生结构数据是从源数据与分析任务中生成的结构化工件。

包括：

- Story Model 版本内容
- PlotArc 版本内容
- Chapter Analysis Memory
- Continuation Memory
- Chapter Plan
- Validation Issue List

### 6.2 规则

- 必须持久化
- 必须与其来源版本和任务绑定
- 可被新版本替代，但不能覆盖历史

### 6.3 建议存储

- 结构化 JSON 存数据库
- 大对象可拆表而不是全部塞单字段

## 7. 运行态数据

### 7.1 定义

运行态数据是系统当前执行中的状态与进度。

包括：

- 任务状态
- 任务进度
- 任务重试计数
- 当前错误
- 暂停/恢复信息
- Agent Run 状态

### 7.2 规则

- 所有长任务的运行态必须持久化
- 进度信息不得只保存在内存
- 任务恢复必须以持久化运行态为依据

### 7.3 建议存储

- 作业表
- 任务表
- 任务事件表

## 8. 可重建视图数据

### 8.1 定义

可重建视图是前端与调试便利使用的派生展示层。

包括：

- Memory View
- 概览页结构摘要
- 作者助手建议展示对象
- 剧情弧展示聚合结果

### 8.2 规则

- 可以持久化，也可以缓存
- 必须可重建
- 不可作为唯一事实来源

### 8.3 建议

- 可保存在数据库中作为快速读取投影
- 重建入口必须独立存在

## 9. 日志与追踪数据

### 9.1 定义

用于问题定位与质量分析。

包括：

- 请求日志
- 任务执行日志
- Agent Run Trace
- 模型调用摘要
- 错误与重试记录

### 9.2 规则

- 至少摘要级别必须持久化
- 必须可按 `request_id / task_id / novel_id / agent_run_id` 查询

## 10. 数据放置原则

### 10.1 必须进数据库

- 小说
- 章节
- 大纲元数据
- 小说状态
- 章节状态
- 任务状态与进度
- Story Model 版本
- PlotArc 版本
- 章节分析记忆
- 续写记忆
- 计划
- 问题单
- 锁定规则
- 当前生效版本指针

### 10.2 可以进文件存储或对象存储

- 超长原文
- 导入原文件副本
- 导出文件
- 调试快照大对象

### 10.3 只适合缓存

- 非关键页面聚合结果
- 热门详情页瞬时投影
- 可快速重算的 UI 辅助数据

## 11. 可重建性矩阵

### 11.1 不可重建，必须保留

- 用户原始正文
- 用户原始大纲
- 用户手工修改内容
- 用户锁定规则
- 发布版本
- 任务历史

### 11.2 可重建，但建议保留

- Memory View
- 展示层摘要
- 部分推荐结果

### 11.3 可丢弃重建

- 临时上下文缓存
- 临时 Prompt 组装中间态
- 页面级聚合缓存

## 12. 存储分层建议

建议最少分四层：

1. 领域数据库层
2. 文件/对象资产层
3. 运行态任务层
4. 缓存与投影层

## 13. 当前系统必须修正的点

重构时必须保证：

1. 整理进度不得只在内存
2. 任务恢复不得依赖前端本地状态
3. 视图数据不得被当作领域事实源
4. 大文本导入与结构结果必须分层存储

## 14. 建议的判断原则

任何一个对象都必须回答五个问题：

1. 它是不是源数据
2. 它是不是领域事实
3. 它是不是运行关键状态
4. 它是否可重建
5. 它是否需要版本化

如果这五个问题答不清，这个对象的存储位置就不应该先实现。
