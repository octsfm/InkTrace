## 一、项目背景

### 1.1 一二期成果

| 期数 | 功能 |
|------|------|
| 一期 | 小说导入、风格分析、续写引擎、一致性检查 |
| 二期 | 多作品管理、模板系统、人设管理、世界观管理 |

### 1.2 核心问题

当前续写流程存在以下问题：

| 问题 | 影响 |
|------|------|
| 上下文有限 | 只能传入最近几章，遗漏早期伏笔 |
| 无法语义检索 | 关键词匹配无法理解"主角上次受伤在哪" |
| 人物一致性差 | 长篇小说容易人设崩坏 |
| 伏笔追踪难 | 无法自动识别和追踪伏笔 |

### 1.3 三期目标

引入**向量数据库 + RAG架构**，实现：
1. **智能检索** - 语义搜索相关章节、人物、伏笔
2. **上下文增强** - 自动组装最相关的上下文给LLM
3. **伏笔追踪** - 自动识别和提醒未回收的伏笔
4. **一致性保障** - 检索相关设定避免矛盾

## 三、技术方案

### 3.1 向量数据库选型

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **ChromaDB** | 轻量、本地、Python原生 | 功能较少 | ✅ 推荐 |
| FAISS | 性能极佳 | 需手动管理 | 备选 |
| Milvus | 功能强大 | 部署复杂 | 不推荐 |

### 3.2 嵌入模型选型

| 方案 | 说明 | 推荐 |
|------|------|------|
| **text2vec-chinese** | 中文优化，本地运行 | ✅ 推荐 |
| BGE | 智源开源，效果好 | 备选 |
| OpenAI Embedding | 效果好但收费 | 不推荐 |

### 3.3 RAG架构

```
┌─────────────────────────────────────────────────────────────┐
│                     向量数据库 (ChromaDB)                     │
│  ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐   │
│  │ 章节向量   │ │ 人物向量   │ │ 世界观向量 │ │ 伏笔向量   │   │
│  └───────────┘ └───────────┘ └───────────┘ └───────────┘   │
└─────────────────────────────────────────────────────────────┘
                          ▲
                          │ 语义检索
                          │
┌─────────────────────────────────────────────────────────────┐
│                     RAG续写引擎                              │
│  1. 用户请求："续写林风进入秘境后的遭遇"                      │
│  2. 向量检索 → 返回：相关章节 + 人物设定 + 伏笔              │
│  3. Prompt组装 → 上下文 + 人物设定 + 世界观 + 请求           │
│  4. LLM调用 → DeepSeek-V3生成新章节                         │
└─────────────────────────────────────────────────────────────┘
```

## 四、验收标准

### 4.1 向量数据库验收

- [ ] ChromaDB成功集成
- [ ] 章节内容可向量化存储
- [ ] 语义搜索返回正确结果
- [ ] 相似度检索功能正常

### 4.2 RAG续写验收

- [ ] 续写时自动检索相关章节
- [ ] 检索内容正确组装到Prompt
- [ ] 续写内容与已有内容一致
- [ ] 长篇小说伏笔不遗漏

### 4.3 性能验收

- [ ] 向量化速度 < 1秒/章
- [ ] 检索响应 < 500ms
- [ ] 内存占用 < 2GB（100万字）
