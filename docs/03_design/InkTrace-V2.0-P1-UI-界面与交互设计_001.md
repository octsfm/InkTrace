## 一、文档定位与设计目标

### 1.1 文档定位

本文档是 InkTrace V2.0-P1 的 UI/UX 详细设计文档，定义 P1 所有页面、工作区、交互流程、视觉规范与信息架构。

本文档是 P1-11（API 与前端集成边界详细设计）的前置输入——先冻结交互与信息架构，P1-11 再根据 UI/UX 设计定义 API / DTO / 前端数据协议。

本文档不替代 P1-01~P1-10 模块内部设计，不写代码，不修改源码，不进入开发计划。

### 1.2 设计目标

1. **写作编辑器永远是主舞台**：正文编辑区占据最大视觉面积，不被任何 AI 面板覆盖、挤压或降级。
2. **AI 进入右侧可折叠工作区**：所有 AI 能力统一收纳到右侧栏的"AI"和"审阅"Tab，不再作为正文下方的长期面板。
3. **三栏布局稳定清晰**：左章节 → 中编辑 → 右资料/AI，职责分明，可独立折叠。
4. **所有门控以卡片/任务流呈现**：DirectionSelection、PlanConfirmation、HumanReviewGate、MemoryReviewGate、ConflictGuard 各自独立、语义清晰、互不混淆。
5. **P0 到 P1 平滑演进**：P0 AI Panel 能力迁移到右侧工作区，不丢失功能，不破坏用户习惯。
6. **用户看的是"能写、能看、能确认"，不是看日志**：调试信息默认折叠，三级信息不可见。

### 1.3 设计原则

| 原则 | 说明 |
|---|---|
| 编辑器优先 | 正文编辑器始终是最大区域，AI 不得覆盖或挤压 |
| 右侧收纳 | AI / 审阅 / 资产统一进入右侧可折叠工作区 |
| 门控独立 | 每种用户确认（Direction / Plan / HumanReview / Memory / Conflict）语义隔离 |
| 卡片任务流 | AI 交互以卡片式任务呈现，非调试控制台 |
| 状态可见 | ready / degraded / blocked / running / waiting 可一眼识别 |
| 信息分层 | 一级（正文/导航）→ 二级（资产/AI状态/门控）→ 三级（Trace/日志，默认折叠） |
| 不提前产品化 | P2 能力（自动连续续写、Style DNA、Citation Link）不进入 P1 UI |
| 安全不透出 | API Key / 完整 Prompt / 完整 ContextPack / 完整 CandidateDraft 不出现在任何 UI 中 |

## 二、当前 P0 界面问题总结

### 2.1 已知问题

| 问题 | 现象 | 根因 |
|---|---|---|
| AI 面板位置不当 | AI Panel 被放在正文编辑区内部或下方，挤压写作空间 | 缺少右侧工作区架构 |
| AI 面板遮挡正文 | AI 面板展开时覆盖/挤压编辑器区域 | 没有三栏独立折叠设计 |
| AI 面板像调试台 | 展示后端状态原文、技术术语过多 | 缺少用户可理解的信息转译层 |
| 资产入口分散 | 大纲/线索/伏笔/人物通过右侧 AssetRail 小图标入口，内容在 Drawer 中展示，与 AI 割裂 | 缺少统一的右侧工作区 Tab 体系 |
| 缺少门控交互 | accept/reject/apply 散落在 API 调用中，没有成体系的交互卡片 | P0 只做了最小 AI 面板 |
| P1 信息量暴增 | AgentSession / Direction / ChapterPlan / CandidateDraftVersion / AIReview / Conflict / MemoryRevision 无处安放 | P1 新能力需要新的 UI 承载层 |

### 2.2 关键不兼容

如果直接将 P1 的 AgentSession、Direction Proposal、ChapterPlan、CandidateDraftVersion、Multi-round Review、ConflictGuard、MemoryReviewGate 塞入当前 P0 AI Panel，将导致：

- AI Panel 变成无法阅读的信息瀑布
- 用户分不清"方向选择"和"候选稿确认"
- Conflict / Memory 警告淹没在 AI 面板日志中
- 正文编辑器更加被挤压

**必须以新架构承载 P1 UI。**

## 三、P1 UI 总体设计原则

### 3.1 核心布局铁律

1. **AI 工作区不得覆盖正文编辑器。**
2. **AI 不得作为正文编辑器下方的大面板长期存在。**
3. **正文编辑器始终是写作台主区域。**
4. **AI 能力统一收纳到右侧可折叠工作区。**
5. **右侧工作区可以展开/收起/切换 Tab。**
6. **弹窗只用于短确认（如 apply 二次确认、版本冲突），不承载长期工作流。**
7. **CandidateDraft / AIReview / ConflictGuard / MemoryReviewGate 不得混在一个无结构的大面板里。**
8. **用户确认门必须明确区分：DirectionSelection ≠ PlanConfirmation ≠ HumanReviewGate ≠ MemoryReviewGate ≠ ConflictGuard。**

### 3.2 P0 → P1 演进约束

- P0 AI Panel 是最小功能入口，P1 迁移到右侧工作区，不再保留为独立面板。
- P0 正文编辑器、章节侧栏、顶部状态栏保留并增强。
- P0 的 AssetRail（右侧资产入口图标）合并入右侧工作区的"大纲/线索/伏笔/人物"Tab。
- P0 的 AssetDrawer 内容迁移到右侧工作区 Tab 内，不再以 Drawer 形式出现。

## 四、整体布局设计

### 4.1 桌面端三栏布局

```text
┌──────────────────────────────────────────────────────────┐
│  顶部状态栏（56px）                                        │
│  作品名 · 初始化状态 · ContextPack readiness · 保存状态      │
├──────────┬────────────────────────┬──────────────────────┤
│          │                        │                      │
│ 左侧     │   中间                  │   右侧               │
│ 章节导航 │   正文编辑器             │   资料与 AI 工作区    │
│          │                        │                      │
│ 260px    │   自适应（主宽度）       │   折叠: 48px         │
│          │                        │   展开: 360px        │
│          │                        │                      │
│          │                        │  [大纲] [线索] [伏笔]  │
│          │                        │  [人物] [AI] [审阅]   │
│          │                        │                      │
├──────────┴────────────────────────┴──────────────────────┤
│  底部状态条（32px）                                        │
│  字数 · 阅读时间 · 章节版本 · 同步状态 · AI 运行指示         │
└──────────────────────────────────────────────────────────┘
```

### 4.2 布局尺寸规范

| 区域 | 桌面端（≥1280px） | 中等屏幕（1024-1279px） | 小屏幕（<1024px） |
|---|---|---|---|
| 顶部状态栏 | 56px 高 | 56px | 48px |
| 左侧章节栏 | 260-300px 宽，可折叠 | 默认折叠，可展开 | 抽屉模式 |
| 中间编辑器 | 自适应剩余宽度 | 自适应，优先保留宽度 | 全宽 |
| 右侧工作区 | 折叠 48px，展开 360px（宽屏可 420px） | 默认折叠，抽屉打开 | 底部 Sheet 或全屏二级页 |
| 底部状态条 | 32px | 32px | 可选隐藏 |

### 4.3 折叠交互

- **左侧章节栏**：点击左侧边缘或顶部汉堡图标折叠为 48px 窄条（仅图标），再次点击展开。编辑器自动扩展。
- **右侧工作区**：点击右侧边缘或顶部 Tab 图标折叠为 48px 窄条（仅图标），再次点击展开。编辑器自动扩展。
- **同时折叠左右**：编辑器占满全宽，进入"沉浸写作模式"。
- **快捷键**：`Ctrl+\` 切换左侧栏，`Ctrl+]` 切换右侧栏，`Ctrl+Shift+F` 进入/退出沉浸模式。

- 撤销 / 重做

- 手动保存

## 七、右侧资料与 AI 工作区设计

### 7.1 Tab 结构

右侧工作区包含 6 个 Tab：

| Tab | 图标（方向） | 职责 | 数据来源 |
|---|---|---|---|
| 大纲 | 📋 | 作品大纲/章节大纲/当前章节目标/ChapterPlan | P0 Outline + P1 ChapterPlan |
| 线索 | 🔗 | 当前章节相关线索/未闭合线索/已回收线索 | StoryMemory.plot_threads |
| 伏笔 | 🪝 | 已埋伏笔/待回收伏笔/风险提示 | StoryMemory + ConflictGuard |
| 人物 | 👤 | 当前出场人物/人物状态/关系/最近变化 | StoryMemory.characters |
| AI | ✦ | AI Settings/初始化/ContextPack/续写/CandidateDraft/QuickTrial/Agent | P0+P1 AI 全能力 |
| 审阅 | 🔍 | AIReview/ConflictGuard/HumanReviewGate/MemoryReviewGate | P0 AIReview + P1 门控 |

### 7.2 折叠态

右侧工作区折叠为 48px 窄条，仅显示 6 个 Tab 图标竖排。当前活跃 Tab 高亮。有新信息时（新的 CandidateDraft / Review 完成 / Conflict 触发）对应 Tab 图标右上角显示小圆点徽标。

### 7.3 展开/收起

- 点击任意 Tab 图标 → 展开右侧栏并切换到该 Tab
- 点击已展开的当前 Tab 图标 → 折叠右侧栏
- 拖动右侧栏左边缘 → 调整宽度（320px ~ 480px）

### 9.2 候选稿状态视觉

| 状态 | 标签颜色 | 操作按钮 |
|---|---|---|
| `pending_review` | 蓝色 | [接受] [拒绝] [查看全文对比] |
| `accepted` | 绿色 ✓ | [应用到草稿] [查看全文对比] |
| `rejected` | 灰色 ✗ | [查看原文]（只读） |
| `applied` | 绿色 ✓✓ | 已应用到第3章 v4 · [查看] |
| `stale` | 橙色 ⚠ | 此候选稿上下文可能过期 · [仍可查看] |

**accepted ≠ applied 的视觉区分**：
- `accepted`：卡片左边缘绿色竖线 + "已接受"标签 + [应用到草稿] 按钮（主操作）
- `applied`：卡片左边缘绿色双线 + "已应用"标签 + 应用到章节信息

### 9.3 全文对比视图

点击 [查看全文对比] 后，右侧工作区切换为对比模式：

```text
┌─ 全文对比 ──────────────────┐
│ [并排] [统一 diff]           │
├─────────────────────────────┤
│ 原文              │ 候选稿   │
│ 顾迟推开...       │ 顾迟推开 │  ← 相同（灰色）
│                   │ 灯塔的   │  ← 新增（绿色）
│ 灯室里那盏灯...    │ 灯室里   │  ← 修改（蓝色）
│                   │ 的光以   │  ← 新增（绿色）
│                   │ 一种他   │  ← 新增（绿色）
└─────────────────────────────┘
```

P1 多轮版本（CandidateDraftVersion）在对比视图中以版本下拉切换：`v1 (原始) | v2 (修订) | v3 (当前)`。

### 9.4 HumanReviewGate 操作流

```text
用户看到候选稿卡片（pending_review）
  │
  ├─ [接受]
  │    └→ 状态变为 accepted
  │       └→ 按钮变为 [应用到草稿]
  │          └→ 用户点击 [应用到草稿]
  │             ├→ apply 需要：idempotency_key + expected_chapter_version
  │             ├→ 版本冲突 → 弹窗显示冲突提示 + [刷新版本] [取消]
  │             ├→ apply 成功 → 卡片状态变为 applied + 编辑器高亮插入内容
  │             └→ apply 失败 → safe_message 提示
  │
  ├─ [拒绝]
  │    └→ 可选填写拒绝理由
  │       └→ 状态变为 rejected
  │          └→ 卡片置灰，只读
  │
  └─ [忽略]（保持 pending_review）
       └→ 无状态变化，稍后处理
```

### 9.5 apply 二次确认弹窗

```text

│                                 │
│ 应用后可通过 V1.1 本地保存        │
│ 链路正常保存。                   │
│                                 │
│ [确认应用]          [取消]       │
└─────────────────────────────────┘
```

冲突级别视觉：
- `blocking`：红色左边框 + ⊘ 图标 + [处理冲突] 为主操作
- `warning`：橙色左边框 + ⚠ 图标 + 允许跳过

### 13.3 硬性规则

- blocking 级别冲突未处理时，[应用到草稿] 按钮置灰 + 提示"请先处理资产冲突"
- ConflictGuard 不提供 [自动修复] 按钮
- 用户必须对每个 blocking 冲突明确选择处理方式

## 十六、视觉风格规范

### 16.1 主风格：Notion 式

- 白底 / 极浅灰背景（`#FFFFFF` / `#FAFAFA` / `#F5F5F5`）
- 柔和卡片（圆角 12-16px，边框 `#E5E5E5`，阴影极轻 `0 1px 3px rgba(0,0,0,0.04)`）
- 大留白（卡片间距 16-20px，段落行高 1.8）
- 文字层级：标题 18-20px / 正文 15-16px / 辅助 13px / 标签 11-12px
- 字体：系统默认中文字体栈（PingFang SC / Microsoft YaHei / Noto Sans SC）

### 16.2 状态流：Linear 式

用于 Agent 进度、AI 任务、ReviewIssue、ConflictGuard：

- 紧凑列表项（高度 36-48px）
- 左侧状态图标 + 标题 + 右侧状态标签
- 标签颜色克制：绿/橙/红/蓝/灰，均为柔色调（饱和度 40-60%）
- 列表可扫描，信息密度适中

### 16.3 资料阅读：Mintlify 式

用于大纲、人物、伏笔、线索、StoryMemory 摘要：

- 文档式排版，段落间距舒适（margin-bottom 12-16px）
- 小标题（16-18px，加粗）→ 正文（15px，常规）
- 信息块可折叠（折叠箭头 + 标题）
- 代码块风格用于展示文本片段（灰底 `#F0F0F0`，等宽字体）

### 16.4 颜色语义

## 十七、信息架构与优先级

### 17.1 三级信息分层

| 层级 | 内容 | 展示位置 | 默认可见 |
|---|---|---|---|
| 一级 | 正文编辑器、章节标题、保存状态、当前章节导航 | 中间 + 左侧 + 顶部 | 始终可见 |
| 二级 | 大纲/人物/伏笔/线索、AI 状态卡片、CandidateDraft、审阅结果、门控操作 | 右侧工作区 | Tab 展开后可见 |
| 三级 | AgentTrace、ToolAudit、LLMCallLog、开发调试信息 | 右侧栏"展开详细 Trace"折叠区内 | **默认折叠** |

### 17.2 用户心智模型

用户写作时的关注顺序：

1. 我在写什么？（正文编辑器 + 章节标题）
2. 写到哪了？（章节导航 + 字数）
3. 故事设定是什么？（大纲 + 人物 + 线索 + 伏笔）
4. AI 能帮我什么？（AI 状态 + CandidateDraft）
5. AI 建议可靠吗？（AIReview + ConflictGuard）
6. AI 具体做了什么？（AgentTrace — 几乎不看）

UI 设计必须支持这个心智模型：1-3 始终可访问，4-5 一触可及，6 深埋。

## 二十、与 P1-11 API 与前端集成边界的关系

### 20.1 设计顺序

1. **P1-UI（本文档）先冻结**交互与信息架构
2. **P1-11** 再根据本文档设计 API / DTO / 前端集成边界
3. P1-11 不能脱离 UI/UX 设计单独定义前端数据

### 20.2 P1-11 需从本文档获取的关键输入

| UI 需求 | P1-11 需定义的 API/DTO |
|---|---|
| AI Tab 状态一览 | `GET /api/v2/ai/dashboard` 聚合接口（settings + init + context_pack + agent 摘要） |
| Agent 进度卡 | `GET /api/v2/ai/sessions/{id}` 含 steps + status + progress |
| 方向选择卡片 | `GET /api/v2/ai/directions` 列表 + `POST /api/v2/ai/directions/{id}/select` |
| ChapterPlan | `GET /api/v2/ai/chapter-plans` + `PUT /api/v2/ai/chapter-plans/{id}` |
| 候选稿卡片 + 对比视图 | `GET /api/v2/ai/candidate-drafts` 扩展 version 列表 + diff |
| 审阅结果卡片 | `GET /api/v2/ai/reviews` 扩展 issues + 分项评分 |
| 冲突横幅 + 处理卡片 | `GET /api/v2/ai/conflicts` + `POST /api/v2/ai/conflicts/{id}/resolve` |
| 记忆更新卡片 | `GET /api/v2/ai/memory-revisions/suggestions` + `POST .../confirm` |
| 轮询 | AgentSession 状态轮询间隔 2-5s |

### 20.3 前端数据协议方向

- 所有卡片返回 `safe_summary` 字段（非完整正文/非完整 Prompt）
- 状态标签统一为枚举字符串
- warning / error 返回 `safe_message` + `error_code`（不返回原始异常堆栈）
- CandidateDraft 列表**默认不返回 content**，详情接口单独返回
- AgentTrace 默认不返回 detail 级别，展开后才加载

## 二十三、P1-UI 待确认点

以下为设计过程中识别的待确认点，不阻塞 P1-UI 文档发布，但建议在进入前端实现前确认：

1. **右侧工作区默认宽度**：建议 360px（可拖拽 320~480px）。最终是 320px、360px 还是 400px？
2. **AI Tab 和审阅 Tab 是否合并**：当前设计分开，AI Tab 管"生成"，审阅 Tab 管"审查+门控"。是否保持分离？
3. **CandidateDraft 预览方式**：当前设计为右侧栏卡片 + 可展开全文 + 对比视图（右侧栏内切换为并排 diff）。是否需要在中间编辑器上方增加临时对比条？
4. **apply 二次确认**：当前设计有弹窗。是否改为右侧栏内确认（不弹窗）以减少打断感？
5. **Direction Proposal 展示形式**：当前设计为垂直卡片列表（A/B/C 堆叠）。是否改为水平横排三列？
6. **ChapterPlan 编辑方式**：当前设计在右侧栏直接展示 + 编辑。是否允许在右侧栏直接编辑章节计划文本？
7. **AgentTrace 可见性**：当前设计默认折叠，开发者模式下可展开。普通用户是否需要看到"当前 Agent 在做什么"的一句话描述？
8. **移动端策略**：当前设计移动端优先保留正文编辑器。是否需要为移动端设计专门的 AI 轻量入口（如底部浮动按钮）？
9. **右侧 Tab 排序**：当前设计顺序为 大纲/线索/伏笔/人物/AI/审阅。是否允许用户自定义 Tab 顺序？
10. **P0 当前 AI Panel 的处理**：设计建议完全移除。是否保留一个"开发者模式"下的原始 AI Panel 入口用于调试？
11. **编辑器内 AI 内容高亮**：当前设计 AI 生成内容插入后以浅蓝色背景高亮。apply 后是否保留高亮（如保留 3 秒后渐隐）还是立即消失？
12. **SSE 事件驱动 vs 轮询**：当前设计默认轮询。P1 是否必须启用 SSE 推送 Agent 状态变更？如果启用，前端如何处理 SSE 断开重连？
