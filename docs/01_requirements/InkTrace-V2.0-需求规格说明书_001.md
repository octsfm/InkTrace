# InkTrace V2.0 需求规格说明书

版本：v2.0-draft  
更新时间：2026-05-07  
范围定位：在 V1.1 非 AI 创作工作台和结构化写作资产的基础上，接入 AI 辅助创作能力，形成长篇小说 AI 写作智能体工作台。  
V2.0 的核心是受控智能体系统，不是单个 AI 按钮，也不是 AI 自动写书工具。

***

## 1. 阶段关系

- V1：稳定纯文本写作底座
- V1.1：非 AI 创作工作台（结构化写作资产沉淀）✅ 已落地
- V2.0：基于结构化资产接入受控 AI 智能体系统（本版本）

***

## 2. 交付批次

- **V2.0-P0**：AI 基础设施 + 核心写作 AI + AI 初始化（必须交付）
- **V2.0-P1**：智能体工作流 + 记忆系统 + 剧情轨道 + 审稿修订（必须交付，后置批次）
- **V2.0-P2**：@引用系统 + 大纲辅助 + 签约开篇助手 + 分析看板（可选交付）

交付说明：
- V2.0-P0 完成后可独立提测
- V2.0-P1 完成后才视为 V2.0 完整完成
- V2.0-P2 是否交付不影响 V2.0 完整性定义

***

## 3. 目标与原则

### 3.1 目标

- AI 带着整本小说的记忆进行续写、改写、审稿和资料整理
- AI 生成的内容必须经过人工确认门控，不自动覆盖任何正式数据
- 多受控智能体协作完成写作流程，不是单个 AI 按钮
- 长篇小说场景下，系统能够维护全书级别的剧情、角色、设定一致性

### 3.2 产品原则

- **Human Review Gate**：AI 可以生成建议和候选稿，但不能自动合并正文、覆盖资产、创建正式章节或改变作品主线
- **Context Pack 驱动**：每次 AI 操作前必须组装完整的上下文情报包
- **异步后处理**：章节保存后异步触发分析管道，不影响正文写作流
- **分层记忆**：章节摘要 → 阶段摘要 → 卷摘要 → 全书进度摘要，四层压缩支撑长篇记忆
- **Agent Trace 可追溯**：每次智能体执行的步骤、输入、输出均记录，可回溯排查

### 3.3 Kimi / DeepSeek 分工原则

- **Kimi**：负责分析、摘要、记忆抽取、规划、写作任务生成、审稿
- **DeepSeek**：负责续写、改写、润色、对白、场景生成、修订
- 业务代码通过 Model Router 按任务角色调用模型，不硬编码 Provider

### 3.4 AI 边界（写硬）

V2.0 明确禁止：

- AI 不自动合并正式正文
- AI 不自动覆盖资产
- AI 不自动创建正式章节
- AI 不改变 V1.1 Local-First 保存链路
- V2.0 不做无人化自动写书

***

## 4. 规则编号体系

- R-AI-INFRA-XX：AI 基础设施（Provider / Prompt / Model Router / Token 追踪）
- R-AI-INIT-XX：作品 AI 初始化（大纲分析 / 正文分析 / 记忆构建）
- R-AI-MEMORY-XX：小说记忆系统（Story Memory / 分层摘要 / 向量索引）
- R-AI-ARC-XX：剧情轨道系统（四层轨道 / 方向推演）
- R-AI-CONTEXT-XX：上下文情报包（Context Pack / Token 优先级）
- R-AI-WRITE-XX：AI 写作（续写 / 改写 / 润色 / 候选稿）
- R-AI-REVIEW-XX：AI 审稿与修订（一致性 / 风格 / AI 味）
- R-AI-AGENT-XX：智能体系统（Agent Workflow / 分工协作）
- R-AI-ASSET-XX：资产 AI 辅助（后处理管道 / AI 建议区 / 冲突保护）
- R-AI-CITE-XX：@标签引用系统
- R-AI-JOB-XX：AI 任务系统
- R-AI-CONFIG-XX：AI 配置与成本
- R-AI-BOUNDARY-XX：AI 边界与禁止

## 5. V2.0-P0（必须）：AI 基础设施 + 核心写作 AI + AI 初始化

### R-AI-INFRA-01 Provider 抽象层

- **Rule**：系统必须提供统一的 AI Provider 抽象，支持 Kimi 和 DeepSeek 两种 Provider，业务代码不直接调用具体 Provider SDK。
- **Behavior**：
  - 定义 `AIProvider` 接口，包含 `chat(messages, options) → Stream`、`count_tokens(text) → int`、`get_model_info() → ModelInfo`。
  - KimiProvider 和 DeepSeekProvider 分别实现该接口。
  - 每个 Provider 实现 token 计价、流式输出、错误重试与超时处理。
- **Edge Case**：
  - Provider 调用失败时，返回结构化错误信息，业务层根据错误类型决定重试或降级。
  - Provider Key 未配置时，对应能力返回"未配置，请在设置中配置 API Key"。
- **Acceptance**：
  - 切换模型配置后，续写和分析分别使用对应 Provider，业务代码无需修改。

### R-AI-INFRA-02 Model Router 模型路由

- **Rule**：业务代码按任务角色调用模型，不硬编码 Provider。路由规则可配置。
- **Behavior**：
  - 预设路由规则：writer → DeepSeek，reviewer / planner / memory_extractor → Kimi。
  - 路由规则可在 AI 设置中调整。
- **Edge Case**：
  - 角色对应的 Provider 不可用时，路由到备用 Provider 或提示用户配置。
- **Acceptance**：
  - Writer 角色调用 DeepSeek，Reviewer 角色调用 Kimi，可通过配置交换角色模型。

### R-AI-INFRA-03 Prompt Registry 提示词模板管理

- **Rule**：Prompt 模板必须与代码分离，独立管理，支持版本和启用状态。
- **Behavior**：
  - 模板按角色和能力分类存储，每条模板包含：prompt key、版本号、输入 schema、输出 schema、模板正文、启用状态。
  - 模板变更不涉及代码发布。
- **Edge Case**：
  - 模板引用不存在的版本时，使用最近可用版本并记录告警。
- **Acceptance**：
  - 修改 Prompt 模板后，下一次 AI 调用使用新模板，无需重启服务。

### R-AI-INFRA-04 AI Job System 任务系统

- **Rule**：支持 AI 长任务的创建、执行、进度查询、取消、失败重试和结果查询。
- **Behavior**：
  - 长任务（大纲分析、正文分析、索引构建等）通过 Job System 异步执行。
  - 前端可查询任务进度（百分比/阶段描述）、取消任务、重试失败步骤。
  - 任务支持暂停/继续（适用于初始化阶段的长时间分析）。
- **Edge Case**：
  - 任务执行中服务重启：重启后正在执行的任务标记为失败，可手动重试。
  - 任务失败时记录失败步骤和错误原因，支持跳过失败项继续执行。
- **Acceptance**：
  - 100 万字正文分析任务可查看进度、可取消、可重试。

### R-AI-INFRA-05 Output Validator 输出校验

- **Rule**：所有结构化 AI 输出必须经过 JSON Schema 校验，保证字段稳定、类型正确、可落库。
- **Behavior**：
  - 每个需要结构化输出的 Prompt 模板关联一个 JSON Schema。
  - 输出校验失败时自动重试（最多 3 次），重试失败后报错并记录原始输出用于排查。
- **Edge Case**：
  - 无法通过校验的输出记录到 Agent Trace，不写入任何数据表。
- **Acceptance**：
  - 修改 Schema 后输出自动按新 Schema 校验。

### R-AI-INFRA-06 AI 调用日志与成本追踪

- **Rule**：每次 AI 调用记录 provider、model、role、prompt tokens、completion tokens、耗时、错误、成本、用户是否采纳。
- **Behavior**：
  - 日志写入 `ai_usage_logs` 表。
  - 前端可查看今日/本周/本月 AI 用量和估算成本。
  - 支持设置月度 Token 预算上限，超限后自动停止非关键 AI 调用。
- **Edge Case**：
  - 成本估算使用 Provider 官方定价，无官方定价时使用配置单价。
- **Acceptance**：
  - 每次 AI 调用有日志可查，成本可追踪。

### R-AI-INFRA-07 Agent Trace 智能体执行轨迹

- **Rule**：记录每次智能体执行的步骤、模型、Prompt、上下文、召回片段、候选稿、审稿结果和用户决策。
- **Behavior**：
  - 每次 Agent 执行生成唯一 trace_id，关联所有步骤日志。
  - 轨迹数据用于排查问题和分析拒绝原因。
  - 前端可查看最近 N 条执行轨迹详情。
- **Edge Case**：
  - 轨迹数据不自动清理，可手动删除或按时间范围清理。
- **Acceptance**：
  - 用户拒绝候选稿后，可在轨迹中查到拒绝理由和候选稿内容。

### R-AI-INIT-01 作品 AI 初始化两阶段流程

- **Rule**：AI 初始化必须分两阶段执行：先分析大纲，再分析正文。大纲分析完成并落库后，才允许正文分析。
- **Behavior**：
  - 第一阶段：分析作品大纲，提取主线、阶段结构、核心冲突、主要人物、势力、世界观、关键伏笔、预期结局方向。
  - 第二阶段：按章节或分段分析已有正文，生成章节摘要、阶段摘要、卷摘要、全书当前进度摘要。
  - 正文分析必须对照大纲，判断正文实际写到了大纲哪个阶段、哪些剧情完成、哪些偏离、哪些伏笔已出现或未出现。
  - 分析过程中显示进度：大纲分析进度、正文切分进度、章节摘要进度、人物提取进度、伏笔提取进度、向量索引构建进度。
- **Edge Case**：
  - 作品无大纲：跳过第一阶段，正文分析只做摘要和基础提取，不判断进度偏离。
  - 正文为空：跳过正文分析，只记录"无正文"状态。
  - 初始化中断后重新开始：已分析完成的部分不重复执行。
- **Acceptance**：
  - 点击初始化后，自动按顺序执行两阶段，每阶段进度可观察。

### R-AI-INIT-02 大纲分析结果落库

- **Rule**：大纲分析完成后，结果必须保存到数据库，作为后续正文分析和续写的总地图。
- **Behavior**：
  - 分析结果包括：主线描述、阶段划分、核心冲突、主要人物列表、势力列表、世界观要点、关键伏笔、预期结局方向。
  - 落库数据独立于 V1.1 work_outline（用户手写大纲保持不动），存入新的 AI 分析结果表。
- **Edge Case**：
  - 用户后续修改大纲后，AI 分析结果标记为"待同步"，提示用户重新分析。
- **Acceptance**：
  - 大纲分析完成后刷新页面，分析结果仍在。

### R-AI-INIT-03 正文分段分析

- **Rule**：正文分析按章节或分段分析前 30 万字、100 万字或更多正文，生成分层摘要。
- **Behavior**：
  - 按章节生成章节摘要（每章 1-3 句）。
  - 按阶段/卷合并章节摘要，生成阶段摘要和卷摘要。
  - 汇总所有摘要，生成全书当前进度摘要。
  - 支持增量分析：新增章节后只分析新增部分。
- **Edge Case**：
  - 单章过长（> 2 万字）时，按段落分段分析后合并摘要。
- **Acceptance**：
  - 正文分析后，每一章都有对应摘要，卷摘要和全书摘要可查看。

### R-AI-INIT-04 正文分析结果落库形成 Story Memory

- **Rule**：正文分析后形成 Story Memory，包括章节摘要、人物状态、事件、时间线、伏笔、设定、地点、风格、剧情进度。
- **Behavior**：
  - 分析结果落入以下数据表或缓存：
    - 章节摘要（chapter_summaries）
    - 角色状态（character_states）
    - 剧情事件（story_events）
    - 时间线事件（timeline_events 建议补充）
    - 伏笔检测结果（foreshadow_candidates）
    - 设定提取（setting_facts）
    - 风格画像（style_dna）
  - 所有分析结果先进入 AI 建议区，用户采纳后才进入正式资产。
- **Edge Case**：
  - 正文修改后已关联的摘要标记为"待更新"。
- **Acceptance**：
  - 分析完成后可在 AI 建议区查看人物、事件、伏笔等抽取结果。

### R-AI-INIT-05 初始化长任务支持

- **Rule**：大纲分析和正文分析要支持开始、暂停、继续、失败重试、跳过失败章节、重新分析某章、重新构建记忆。
- **Behavior**：
  - 使用 R-AI-INFRA-04 Job System 执行。
  - 用户可在初始化过程中暂停并保存当前进度。
  - 单章分析失败可跳过，不影响后续章节分析。
- **Edge Case**：
  - 跳过章节后，可在初始化完成后手动触发该章节的单独分析。
- **Acceptance**：
  - 初始化过程中可暂停、可重试失败项、可跳过。

### R-AI-INIT-06 初始化完成后才开放续写

- **Rule**：只有大纲分析、正文分析、Story Memory、向量索引、当前故事状态全部完成后，才开放续写能力。
- **Behavior**：
  - 续写按钮在初始化完成前为禁用状态，hover 提示"AI 初始化未完成"。
  - 初始化完成后，续写入口激活。
- **Edge Case**：
  - 作品有新章节发布后，Story Memory 标记为"待同步"，续写仍可用但提示"部分记忆可能未更新"。
  - 用户跳过初始化：允许，但不提供 AI 续写能力，仅提供基础 AI 功能。
- **Acceptance**：
  - 新作品必须先初始化才能使用续写；初始化进度不足时续写入口不可用。

### R-AI-WRITE-01 单章续写

- **Rule**：系统支持根据选定方向、章节计划和 Context Pack 生成下一章候选稿。AI 生成的正文不得直接进入正式正文。
- **Behavior**：
  - 用户在章节编辑器中触发 AI 续写。
  - 系统组装 Context Pack， Planner Agent 生成 Writing Task，Writer Agent 生成候选稿。
  - 候选稿写入候选稿区，不修改正式正文。
  - 候选稿支持流式展示（实时看到生成过程）。
- **Edge Case**：
  - 续写中断：已生成的候选稿部分保留，可丢弃或继续生成。
- **Acceptance**：
  - 点击续写后流式生成内容，候选稿不进入正式正文。

### R-AI-WRITE-02 多章续写

- **Rule**：系统支持用户指定续写 3 章、5 章或续写到当前小剧情结束，并按章生成候选稿、审稿、更新记忆。
- **Behavior**：
  - 用户在续写设置中选择目标章数或"到当前剧情结束"。
  - 系统逐章执行：生成 → 审稿 → 修订 → 展示候选稿。
  - 每章候选稿独立展示，用户逐章确认。
- **Edge Case**：
  - 某章连续修订失败 3 次：暂停自动续写，提示用户介入。
- **Acceptance**：
  - 用户选择续写 3 章后，系统生成 3 章候选稿，每章独立接受/拒绝。

### R-AI-WRITE-03 自动连续续写

- **Rule**：系统支持受控自动续写队列，在用户设定目标章数、字数、停止条件后，自动逐章生成候选稿。
- **Behavior**：
  - 用户设定：目标章数、目标字数、最大 token 消耗。
  - 自动逐章生成，每章生成后执行审稿和修订。
  - 停止条件（任一满足即停止）：
    - 达到指定章数
    - 达到指定字数
    - 剧情阶段结束
    - 审稿发现严重冲突
    - 伏笔提前揭示
    - 连续修订失败超过阈值
    - 达到成本上限
    - 模型调用连续失败
    - 用户手动停止
- **Edge Case**：
  - 自动续写停止后，已生成的章候选稿仍保留在候选稿区。
- **Acceptance**：
  - 设定续写 5 章后，自动逐章生成并停止在第 5 章。

### R-AI-WRITE-04 选区改写/润色

- **Rule**：用户选中正文片段后，可执行扩写、重写、缩写、润色、对白优化、降低 AI 味等操作。
- **Behavior**：
  - 选中文本后弹出浮动工具栏，列出可用改写操作。
  - 选择操作后，AI 生成改写候选稿，在预览浮窗中展示（不直接替换正文）。
  - 用户点击"应用"替换原文，点击"跳过"关闭预览。
  - 对白优化专门处理对话段落，使其更自然符合人物性格。
- **Edge Case**：
  - 选中文本为空或仅含空白：禁用改写按钮。
  - 选中文本超过 5000 字：提示"选中内容过长，建议分段改写"。
- **Acceptance**：
  - 选中正文 → 选择扩写 → 预览 → 应用/跳过，全流程可用。

### R-AI-WRITE-05 Candidate Draft 候选稿机制

- **Rule**：AI 生成的正文不得直接进入正式正文，必须进入候选稿区，由用户接受、插入、替换、丢弃或重新生成。
- **Behavior**：
  - 候选稿区展示：来源（续写/改写/润色）、生成时间、字数、审稿摘要。
  - 用户操作：接受（合并到正式正文）、插入（在光标处插入）、替换（替换选中区域）、丢弃、重新生成。
  - 候选稿在候选稿编辑器中可手动编辑，编辑后再确认合并。
  - 合并后的正文进入正常的 Local-First 保存链路。
- **Edge Case**：
  - 候选稿合并后版本冲突：走现有 409 冲突处理流程。
  - 候选稿被丢弃但用户后悔：提供"最近丢弃"列表，保留 24 小时。
- **Acceptance**：
  - 候选稿可接受、可编辑后接受、可丢弃、可重新生成。

### R-AI-WRITE-06 候选稿多轮迭代

- **Rule**：支持用户基于候选稿反馈重新生成，例如"节奏加快""多用短句""减少心理描写"。
- **Behavior**：
  - 用户在候选稿区输入文本反馈或选择预设反馈标签。
  - 系统携带用户反馈重新生成新版本候选稿。
  - 保留候选稿版本关系：v1、v2、v3，用户可回溯版本。
- **Edge Case**：
  - 连续 5 次反馈迭代后仍不满意：提示建议用户手动编辑。
- **Acceptance**：
  - 在候选稿中输入"节奏加快"后，重新生成的候选稿节奏明显变化。

### R-AI-WRITE-07 Candidate Draft 拒绝理由追踪

- **Rule**：用户拒绝候选稿时，提供轻量的理由选择，写入 Agent Trace。
- **Behavior**：
  - 用户点击"丢弃"时弹出可选理由：方向不对 / 风格不一致 / 设定冲突 / 质量问题 / 其他。
  - 选择后记录到 Agent Trace，不强制选择。
- **Edge Case**：
  - 不选择理由也能丢弃。
- **Acceptance**：
  - 拒绝候选稿后可在执行轨迹中查到拒绝理由。

## 6. V2.0-P1（必须）：智能体工作流 + 记忆系统 + 剧情轨道 + 审稿修订

### 6.0 Agent Workflow 智能体工作流

标准流程：

```
Memory Agent → Planner Agent → Writer Agent → Reviewer Agent → Rewriter Agent → 用户确认 → Memory Agent 更新建议
```

| Agent | 职责 |
|---|---|
| Memory Agent | 大纲分析、章节摘要、人物状态、剧情事件、伏笔、设定变化、时间线、风格特征提取与更新建议 |
| Planner Agent | 分析当前进度，生成后续剧情方向、章节计划、Writing Task、A/B/C 分支推演和续写约束 |
| Writer Agent | 根据 Context Pack 和 Writing Task 生成续写正文、整章候选稿、选区改写、扩写、润色、对白优化 |
| Reviewer Agent | 检查候选稿的人物一致性、设定冲突、时间线冲突、伏笔误用、风格漂移、AI 味、任务完成度 |
| Rewriter Agent | 根据审稿问题对候选稿进行局部修订、降 AI 味、调整节奏、强化钩子和悬念 |

### R-AI-AGENT-01 Memory Agent 记忆智能体

- **Rule**：Memory Agent 负责大纲分析、章节摘要、人物状态、剧情事件、伏笔、设定变化、时间线、风格特征提取与更新建议。
- **Behavior**：
  - 初始化阶段：执行两阶段分析（大纲 + 正文），输出 Story Memory。
  - 运行阶段：用户保存章节后异步触发，生成后处理更新建议。
  - 更新建议写入 AI 建议区，不直接覆盖资产。
- **Edge Case**：
  - Story Memory 更新时发现与用户手动维护的资产冲突：触发 Conflict Guard。
- **Acceptance**：
  - 用户保存章节后，Memory Agent 异步生成更新建议，可在 AI 建议区查看。

### R-AI-AGENT-02 Planner Agent 规划智能体

- **Rule**：Planner Agent 负责分析当前进度，生成后续剧情方向、章节计划、Writing Task、A/B/C 分支推演和续写约束。
- **Behavior**：
  - 用户在续写前可触发方向推演，Planner 基于四层剧情轨道生成 A/B/C 多个后续剧情方向。
  - 每个方向包含：剧情概要、主要冲突、伏笔使用、风险点、未来 3-5 章预估。
  - 用户选择方向后，Planner 生成后续 3-5 章章节计划，包括每章目标、关键事件、冲突推进、伏笔安排和禁止事项。
  - 每次续写前，Planner 生成 Writing Task。
- **Edge Case**：
  - 方向推演时用户已有明确想法：允许用户输入自定义方向，Planner 据此细化。
- **Acceptance**：
  - 续写前可以看到 A/B/C 三个方向概览并选择。

### R-AI-AGENT-03 Writer Agent 写作智能体

- **Rule**：Writer Agent 负责根据 Context Pack 和 Writing Task 生成续写正文、整章候选稿、选区改写、扩写、润色、对白优化。
- **Behavior**：
  - 使用 DeepSeek 模型（可配置）。
  - 流式输出候选稿内容。
  - 生成完成后输出传递给 Reviewer Agent。
- **Edge Case**：
  - 生成内容被截断（达到 max tokens）：标记截断位置，用户可继续生成。
- **Acceptance**：
  - Writer Agent 生成的候选稿符合 Writing Task 要求。

### R-AI-AGENT-04 Reviewer Agent 审稿智能体

- **Rule**：Reviewer Agent 负责检查候选稿的人物一致性、设定冲突、时间线冲突、伏笔误用、风格漂移、AI 味、任务完成度。
- **Behavior**：
  - DeepSeek 生成候选稿后，由 Reviewer Agent（Kimi）检查。
  - 审查维度包括：
    - 人物一致性：角色行为是否符合人物卡设定和当前状态
    - 设定冲突：是否违反作品世界观设定
    - 时间线冲突：事件顺序是否合理
    - 伏笔误用：是否错误使用了未揭示的伏笔
    - 风格漂移：语言风格是否偏离作品基线
    - AI 味：检测典型 AI 生成特征
    - 任务完成度：是否完成 Writing Task 中设定的目标
  - 审查结果分为：通过 / 警告（轻微问题）/ 不通过（严重问题）。
- **Edge Case**：
  - 审查不通过次数 > 3 次：暂停工作流，提示用户介入。
- **Acceptance**：
  - 候选稿生成后自动触发审稿，审稿结果对用户可见。

### R-AI-AGENT-05 Rewriter Agent 修订智能体

- **Rule**：Rewriter Agent 负责根据审稿问题对候选稿进行局部修订、降 AI 味、调整节奏、强化钩子和悬念。
- **Behavior**：
  - 审稿结果为"不通过"且为可自动修复的问题时，自动触发修订。
  - 修订后自动进入下一轮审稿。
  - 修订最多执行 3 轮，超过后标记为"需要人工处理"。
- **Edge Case**：
  - 修订轮次耗尽后，候选稿标记具体问题并保留给用户手动修改。
- **Acceptance**：
  - 候选稿发现设定冲突后，修订版本自动修复该冲突。

### R-AI-AGENT-06 Opening Agent 签约开篇智能体

- **Rule**：Opening Agent 负责分析平台参考小说前 1-3 章，总结开篇规律，并辅助生成用户原创前三章候选稿。
- **Behavior**：
  - 用户导入某平台多部参考小说前 1-3 章。
  - Opening Agent 总结：开篇钩子模式、节奏、冲突引入方式、爽点密度、章节结尾悬念手法、平台文感特征。
  - 基于分析结果，辅助生成用户原创前三章候选稿。
  - 对前三章候选稿执行签约向审稿：开篇钩子强度、主角目标清晰度、冲突提前量、爽点密度、章节结尾悬念、过度模仿风险。
- **Edge Case**：
  - 用户不导入参考小说：Opening Agent 跳过分析阶段，直接基于通用规则辅助生成。
- **Acceptance**：
  - 导入参考小说后可查看平台开篇规律分析。

***

### R-AI-MEMORY-01 Story Memory 小说长期记忆

- **Rule**：系统维护章节摘要、阶段摘要、卷摘要、全书摘要、角色卡、角色状态、设定、事件、伏笔、风格画像，构成小说长期记忆。
- **Behavior**：
  - 记忆数据分层存储，每层有独立的 token 估算和裁剪策略。
  - 记忆数据可通过 API 查询和手动修正。
  - 记忆数据随正文更新而增量更新，不每次全量重建。
- **Edge Case**：
  - 用户手动修改正文后，相关记忆标记为"待更新"。
- **Acceptance**：
  - 正文 100 章后，Story Memory 仍可在 2 秒内查询。

### R-AI-MEMORY-02 Story State 当前故事状态锁

- **Rule**：系统记录当前地点、角色状态、境界/能力、阵营关系、当前剧情阶段、未解伏笔、禁止事项，形成故事状态锁，防止 AI 写崩。
- **Behavior**：
  - Story State 在每次记忆更新时同步刷新。
  - Writer Agent 生成候选稿时，Context Pack 携带 Story State。
  - Reviewer Agent 审稿时对照 Story State 检查违规。
  - 用户可手动编辑 Story State 中的禁止事项字段。
- **Edge Case**：
  - Story State 与正文实际状态不一致时，以用户手动修正为准。
- **Acceptance**：
  - 在 Story State 中标记"主角当前在青云山"，AI 续写不会无故改变地点。

### R-AI-MEMORY-03 Recursive Summary 分层摘要

- **Rule**：系统支持章节摘要 → 阶段摘要 → 卷摘要 → 全书当前进度摘要，用压缩地图支撑长篇续写。
- **Behavior**：
  - 章节摘要：每章 1-3 句，包含关键事件和角色变化。
  - 阶段摘要（10-20 章）：由章节摘要合并压缩生成。
  - 卷摘要（阶段 * N）：由阶段摘要进一步压缩生成。
  - 全书当前进度摘要：最高层压缩，记录故事当前走到哪里。
- **Edge Case**：
  - 阶段边界不明确时，按每 15 章自动分组作为阶段。
- **Acceptance**：
  - 100 章正文的分层摘要可在 500 tokens 内概括全书进度。

### R-AI-MEMORY-04 RAG / Vector Recall 历史剧情召回

- **Rule**：系统通过本地 Embedding + Vector DB，从长篇正文中召回与当前续写相关的旧剧情片段。
- **Behavior**：
  - 正文切片（每片 512 tokens，重叠 64 tokens），Embedding 后入库。
  - 续写前以当前章节最后 500 字和 Writing Task 关键字作为查询，召回 Top-K 相关片段。
  - 召回片段进入 Context Pack 的 RAG 层。
  - 支持按需重建索引。
- **Edge Case**：
  - 正文修改后对应的切片重新 Embedding，原切片标记失效。
- **Acceptance**：
  - 续写第 50 章时，AI 能正确引用第 10 章埋下的伏笔。

### R-AI-MEMORY-05 Embedding Index Pipeline 向量索引管道

- **Rule**：支持章节切片、Embedding、本地向量入库、索引更新、索引重建，并记录 chunk 与章节位置关系。
- **Behavior**：
  - 初始化时全量构建索引。
  - 新增或修改正文时增量更新索引。
  - 每个 chunk 记录来源章节 ID、起始偏移、结束偏移。
- **Edge Case**：
  - 索引构建中服务重启：已构建部分有效，继续未完成部分。
- **Acceptance**：
  - 新增一章正文后，该章内容可在 RAG 召回中命中。

### R-AI-MEMORY-06 结构化故事事实

- **Rule**：系统轻量记录角色关系、阵营关系、物品归属、地点事件、伏笔状态等，不做复杂知识图谱。
- **Behavior**：
  - 数据模型保持扁平，不引入图数据库。
  - 角色关系存储为 JSON array（如 `[{"target": "张三", "type": "师徒", "description": "授业恩师"}]`）。
  - 阵营关系、物品归属等同理。
- **Edge Case**：
  - 关系数据由 AI 提取建议 + 用户确认，不自动抽取。
- **Acceptance**：
  - 可在角色面板查看该角色与其他角色的关系列表。

### R-AI-ARC-01 四层剧情轨道系统

- **Rule**：系统维护全文弧、卷/大段落弧、剧情波次/小段落弧、临近窗口四层轨道。
- **Behavior**：
  - 全文弧（Master Arc）：全书终极目标、最终冲突、最终 BOSS、主角长期动机、终局伏笔、当前主线阶段。
  - 卷/大段落弧（Volume / Act Arc）：当前卷核心矛盾、阶段目标、主要敌人、高潮节点、收束条件、当前阶段位置。
  - 剧情波次/小段落弧（Sequence Arc）：10-20 章级别小剧情循环，包含目标、冲突、关键事件、反转点、爽点、预计收束章节。
  - 临近窗口（Immediate Window）：前 10 章摘要、前 3 章精简内容、当前章节上下文、角色情绪、动作轨迹、场景细节、上一章钩子。
- **Edge Case**：
  - 作品不分卷：省略卷弧层，仅保留全文弧 + 小段落弧 + 临近窗口。
- **Acceptance**：
  - 初始化完成后可在面板中查看四层轨道内容。

### R-AI-ARC-02 剧情轨道落库

- **Rule**：分析完成后，剧情弧和四层轨道需要落库，并记录层级、状态、目标、冲突、章节范围、下一步推进方向。
- **Behavior**：
  - 存入 `story_arcs` 表，包含字段：层级（master/volume/sequence）、状态（active/completed/pending）、目标描述、核心冲突、关联章节范围、下一步推进方向。
  - 用户可手动调整轨道内容。
- **Edge Case**：
  - 用户新增章节超出当前弧范围时，自动扩展弧的章节范围。
- **Acceptance**：
  - 四层轨道内容在刷新后保持，可手动编辑。

### R-AI-ARC-03 剧情轨道进入 Context Pack

- **Rule**：每次续写前，Context Pack 必须包含全文弧、卷弧、小段落弧、临近窗口。
- **Behavior**：
  - 四层轨道按 Token 优先级规则裁剪后并入 Context Pack。
  - 轨道信息影响 Writing Task 的生成（弧的当前阶段决定续写方向）。
- **Edge Case**：
  - 轨道信息为空（未初始化）：Context Pack 跳过轨道层。
- **Acceptance**：
  - Context Pack 中包含四层轨道信息。

### R-AI-ARC-04 剧情轨道 Token 优先级

- **Rule**：Context Pack 超限时，按以下优先级从高到低裁剪：临近窗口 → 当前小段落弧 → 当前卷弧 → 全文弧。全文弧可以短，但不能缺。
- **Behavior**：
  - 临近窗口：完整保留。
  - 当前小段落弧：优先保留目标和冲突，裁剪背景说明。
  - 当前卷弧：优先保留当前阶段位置和收束条件。
  - 全文弧：压缩到 200 tokens 以内的摘要，不可完全移除。
- **Edge Case**：
  - 四层轨道总 token 仍超限：优先压缩全文弧。
- **Acceptance**：
  - Context Pack 总 token 可控制在配置上限内。

### R-AI-ARC-05 剧情轨道与续写模式联动

- **Rule**：续写模式选项受当前剧情轨道状态影响。
- **Behavior**：
  - 当小段落弧即将收束时，自动推荐"续写到本弧结束"。
  - 当卷弧高潮节点接近时，自动推荐"目标字数翻倍"。
- **Edge Case**：
  - 轨道信息不足时只提供基础续写选项。
- **Acceptance**：
  - 切换续写模式时，轨道信息影响推荐选项。

### R-AI-ARC-06 剧情轨道与审稿联动

- **Rule**：审稿时必须检查候选稿是否偏离全文弧、当前卷弧、小段落弧、临近窗口。
- **Behavior**：
  - Reviewer Agent 将候选稿内容与四层轨道对比。
  - 偏离检测结果纳入审稿报告。
- **Edge Case**：
  - 用户有意偏离轨道：允许，但审稿报告标记"已偏离轨道，建议确认"。
- **Acceptance**：
  - 候选稿与轨道方向冲突时，审稿报告给出明确警告。

### R-AI-ARC-07 后续方向推演

- **Rule**：系统基于大纲和已有正文记忆，生成 A/B/C 多个后续剧情方向。
- **Behavior**：
  - 每个方向包含：剧情概要、主要冲突、伏笔使用、风险点和未来 3-5 章预估。
  - A/B/C 方向基于四层剧情轨道生成，不能凭空发散。
  - 方向推演是沙盒推演的一部分，用户不确认则不产生任何候选稿。
- **Edge Case**：
  - 推演时用户已有自定义方向：用户可输入方向文本，Planner 基于该方向细化。
- **Acceptance**：
  - 点击方向推演后，可见 A/B/C 三个不同方向的概览。

### R-AI-ARC-08 用户选择剧情方向

- **Rule**：用户从 A/B/C 后续方向中选择一个方向，系统再基于该方向生成章节计划和下一章 Writing Task。
- **Behavior**：
  - 选择方向后，Planner Agent 生成后续 3-5 章章节计划。
  - 章节计划包括：每章目标、关键事件、冲突推进、伏笔安排和禁止事项。
  - 生成 Writing Task 进入续写阶段。
- **Edge Case**：
  - 用户对生成的章节计划不满意：可返回重新选择方向或调整计划。
- **Acceptance**：
  - 选择方向后可看到 3-5 章的章节计划，每章目标和事件明确。

***

### R-AI-CONTEXT-01 Context Pack 上下文情报包

- **Rule**：每次 AI 续写、改写、审稿前，系统组装包含当前章节、摘要、状态、角色、设定、伏笔、召回片段、写作任务的上下文包。
- **Behavior**：
  - Context Pack 分为以下层级：
    1. 当前章节/选区上下文（最高优先级）
    2. 当前卷摘要和最近 3 章摘要
    3. Story State（当前地点、角色状态、禁止事项）
    4. Writing Task
    5. 全书摘要 + 角色卡 + 设定
    6. RAG 召回片段（按相关性排序）
    7. Style DNA
  - 按 Token 优先级规则裁剪。
  - Context Pack 记录到 Agent Trace。
- **Edge Case**：
  - 不同 AI 操作组装的 Context Pack 层不同（续写包含全部层，改写只包含选区上下文 + 角色卡 + Style DNA）。
- **Acceptance**：
  - 查看 Agent Trace 可看到每次 AI 调用使用的 Context Pack 内容。

### R-AI-CONTEXT-02 Context Pack Token 优先级规则

- **Rule**：上下文超限时按优先级从高到低裁剪，保证关键信息的完整性。
- **Behavior**：
  - 优先级顺序：当前章节/选区上下文 > 当前卷摘要和最近章节摘要 > Story State > Writing Task > 全书摘要/角色卡/设定 > RAG 召回片段 > Style DNA。
  - 每个层级有独立的 token 上限，超限时该层级内部裁剪。
  - RAG 召回片段优先保留与当前上下文语义相似度最高的片段。
- **Edge Case**：
  - 整体 Context 仍超限时，丢弃优先级最低的层级。
- **Acceptance**：
  - Context Pack 总 token 不超过模型上限的 80%。

### R-AI-CONTEXT-03 Context Pack 组装 API

- **Rule**：后端提供统一的 Context Pack 组装服务，所有需要上下文的 AI 能力共用。
- **Behavior**：
  - 输入：work_id、operation_type（continue_writing / rewrite / review / plan）、当前 chapter_id（可选）、选区范围（可选）。
  - 输出：组装好的上下文文本、各层级 token 明细、是否发生过裁剪。
- **Edge Case**：
  - 一次性请求多章 Context Pack（多章续写场景）：按章独立组装。
- **Acceptance**：
  - 不同操作类型返回不同层级的上下文。

### R-AI-CONTEXT-04 Writing Task 写作任务

- **Rule**：正式续写前，系统生成本次写作目标、必须包含内容、禁止内容、情绪基调、目标字数。
- **Behavior**：
  - Writing Task 由 Planner Agent 生成。
  - 字段：目标描述、必须包含的关键元素、禁止内容列表、情绪基调、目标字数。
  - Writing Task 进入 Context Pack，Writer Agent 严格遵循。
  - Reviewer Agent 审稿时检查候选稿是否符合 Writing Task。
- **Edge Case**：
  - 用户对 Writing Task 不满意：可手动编辑 Writing Task。
- **Acceptance**：
  - 每次续写可见本次 Writing Task，候选稿完成后审稿报告包含任务符合度评估。

### R-AI-CONTEXT-05 Attention Filter 动态关注度过滤

- **Rule**：系统根据当前章节、选区、写作任务动态过滤无关角色、地点、设定和召回片段，减少 Context Pack 污染。
- **Behavior**：
  - 根据当前章节上下文判断哪些角色、地点、设定当前活跃。
  - 不活跃的角色卡不进入 Context Pack（除非当前写作任务明确要求）。
  - 召回片段按相关性排序，只取 Top-K。
- **Edge Case**：
  - 过滤过于激进导致遗漏关键信息：保留全文弧摘要作为兜底。
- **Acceptance**：
  - Context Pack 中不包含当前不需要的角色卡。

### R-AI-CONTEXT-06 Sandbox Mode 沙盒推演

- **Rule**：正式续写前，AI 可生成 A/B/C 多个剧情走向摘要，用户选择方向后再生成正式候选稿。
- **Behavior**：
  - 沙盒模式输出：每个走向的 200-300 字剧情概要和核心冲突说明。
  - 用户选择方向后，Planner Agent 生成详细章节计划。
  - 沙盒推演不产生任何候选稿，不消耗正文章节号。
- **Edge Case**：
  - 用户跳过沙盒模式：直接使用默认方向（当前剧情自然延伸）。
- **Acceptance**：
  - 沙盒模式中三个走向概览差异明显，选择后可进入章节计划。

### R-AI-CONTEXT-07 Style DNA 文风指纹

- **Rule**：用户可上传或指定标杆正文，系统提取词频、句式、段落节奏、对白比例、心理描写比例、叙述视角等文风特征，用于约束续写和改写。
- **Behavior**：
  - 用户指定一篇或多篇正文作为风格参考。
  - 系统提取：高频词分布、平均句长、段落长度分布、对白占比、心理描写占比、叙述视角（第一/第三人称）。
  - Style DNA 进入 Context Pack 的底层，约束 Writer Agent 输出风格。
  - 续写和改写的目标是与 Style DNA 风格一致，非克隆。
- **Edge Case**：
  - 未配置 Style DNA：跳过此层，不约束风格。
- **Acceptance**：
  - 配置 Style DNA 后，续写风格明显更接近参考文本。

***

### R-AI-REVIEW-01 AI Review 候选稿审稿

- **Rule**：DeepSeek 生成候选稿后，由 Kimi Reviewer Agent 检查候选稿是否违反设定、人物状态、时间线、伏笔、写作任务和文风约束。
- **Behavior**：
  - 候选稿生成后自动触发审稿。
  - 审查维度：人物一致性、设定冲突、时间线冲突、伏笔误用、风格漂移、AI 味、任务完成度。
  - 审稿结果结构化输出，包含每个维度的评估和问题列表。
  - 审稿结果对用户可见（候选稿区展示审稿摘要）。
- **Edge Case**：
  - 审稿调用超时或失败：标记为"未审稿"，候选稿仍可用。
- **Acceptance**：
  - 候选稿生成后 10 秒内显示审稿结果。

### R-AI-REVIEW-02 签约向审稿

- **Rule**：对前三章候选稿检查开篇钩子、主角目标、冲突提前量、爽点密度、章节结尾悬念、平台文感、过度模仿风险。
- **Behavior**：
  - Opening Agent 生成前三章候选稿后执行签约向审稿。
  - 审查维度在上述基础上增加：市场匹配度评估、与参考小说的差异化建议。
- **Edge Case**：
  - 未导入参考小说时跳过平台文感和过度模仿检查。
- **Acceptance**：
  - 前三章候选稿有签约向审稿报告。

### R-AI-ASSET-01 章节后处理管道

- **Rule**：用户保存章节后（收到 200 OK），系统异步分析章节，生成摘要、事件、角色变化、伏笔候选、设定变化等建议。
- **Behavior**：
  - 触发时机：章节保存 200 OK 后异步执行。
  - 分析产出：章节摘要、本章关键事件列表、角色提及变化、伏笔候选、设定变化候选。
  - 所有结果写入 AI 建议区，不直接写数据库。
  - 管道执行不阻塞用户继续写作。
- **Edge Case**：
  - 管道执行失败：记录失败日志，用户可手动触发重试。
  - 用户连续快速保存多章：管道按保存顺序排队串行执行。
- **Acceptance**：
  - 保存章节后，可在 AI 建议区看到本章分析结果。

### R-AI-ASSET-02 AI 建议区

- **Rule**：AI 后处理得到的人物、事件、伏笔、设定、时间线、摘要等结果，先进入 AI 建议区，用户采纳后才进入正式资产。
- **Behavior**：
  - AI 建议区作为 Asset Drawer 的一个独立 Tab。
  - 建议按类型分类：人物建议、事件建议、伏笔候选、设定变化、时间线建议。
  - 每条建议有单独的操作：采纳、编辑后采纳、忽略。
  - 采纳后写入对应资产表，进入正常 dirty → save 链路。
  - 建议支持过期机制：超过 7 天未处理的建议自动标记为过期。
- **Edge Case**：
  - 建议对应的资产已被用户手动修改：显示对比，用户决定是否覆盖。
- **Acceptance**：
  - AI 建议区可看到分类建议列表，采纳后对应资产更新。

### R-AI-ASSET-03 Conflict Guard 资产冲突保护

- **Rule**：当 AI 建议更新用户手动维护过的角色、设定、伏笔、时间线等资产时，必须显示对比界面，不能自动覆盖。
- **Behavior**：
  - 检测条件：AI 建议的目标资产有用户手动修改记录。
  - 触发后展示对比：左侧用户当前值、右侧 AI 建议值。
  - 用户选择：采纳 AI 建议 / 保留当前值 / 编辑后保存。
- **Edge Case**：
  - 无法判断是否为用户手动修改时（初始状态）：默认不触发冲突保护。
- **Acceptance**：
  - AI 建议与用户手动资产冲突时显示对比界面。

### R-AI-ASSET-04 AI Suggestion Store 建议数据模型

- **Rule**：AI 生成的人物、事件、伏笔、设定、时间线等建议，需要使用统一的状态管理。
- **Behavior**：
  - 建议统一保存状态：pending / accepted / rejected / edited / expired。
  - 每条建议记录来源（后处理管道 / 初始化分析）、生成时间、关联章节 ID。
  - 过期策略：超过 7 天自动标记 expired。
- **Edge Case**：
  - 建议被采纳后源数据被删除：建议状态统一转 rejected。
- **Acceptance**：
  - 建议列表中的每条都有明确的状态标记。

### R-AI-ASSET-05 大纲辅助

- **Rule**：支持大纲润色、大纲扩写、章节细纲建议、下一章写作任务建议。
- **Behavior**：
  - 大纲润色：对现有作品大纲做语言润色和条理化，不改变内容。
  - 大纲扩写：把简短提纲扩展为更详细的大纲。
  - 章节细纲建议：根据作品大纲和已写章节生成下一章细纲建议。
  - 下一章写作任务建议：根据当前进度生成下一章的 Writing Task 建议。
  - 以上所有结果进入 AI 建议区，用户确认后不自动保存。
- **Edge Case**：
  - 作品大纲为空时，大纲润色和扩写禁用，提示"请先输入大纲内容"。
- **Acceptance**：
  - 点击大纲润色后，大纲内容优化但主题不变。

## 7. V2.0-P2（可选）：@引用系统 + 签约开篇 + 分析看板

### R-AI-CITE-01 @标签引用系统

- **Rule**：正文支持 `@人物`、`@地点`、`@事件`、`@伏笔` 等引用，打通正文与 V1.1 资产。
- **Behavior**：
  - 在正文 textarea 输入 `@` 时弹出联想菜单，匹配人物/地点/事件/伏笔名称。
  - 选中的 `@实体` 在正文中高亮显示。
  - 鼠标悬停 `@实体` 时弹出浮窗显示实体摘要（从 Asset 读取）。
  - AI 后处理结果也可以生成 `@` 引用建议，进入 AI 建议区。
  - 保存章节时自动提取所有 `@` 引用并调用 mentions API（存入 `chapter_mentions` 表）。
- **Edge Case**：
  - 被引用的资产被删除：正文中的 `@` 标记保持显示但 hover 提示"已删除"。
- **Acceptance**：
  - 输入 `@` 弹出联想，选择后正文高亮，悬停可查看摘要。

### R-AI-CITE-02 Citation Link 引用追踪

- **Rule**：AI 候选稿中若使用了前文设定、伏笔、角色状态或历史事件，应标记来源，用户可点击查看引用章节或资产。
- **Behavior**：
  - Writer Agent 生成候选稿时标记引用来源。
  - 引用在候选稿中以脚注形式展示：`[1] 第 3 章 - 张三获得青云剑`。
  - 用户点击引用跳转到对应章节或资产。
- **Edge Case**：
  - 引用章节已不存在：标记为"未知来源"，不阻断候选稿使用。
- **Acceptance**：
  - 候选稿中包含引用标记，可查看来源位置。

### R-AI-CONFIG-01 AI 设置

- **Rule**：支持 Kimi Key、DeepSeek Key、连接测试、模型角色配置、temperature、max tokens、timeout、预算、自动分析开关、本地 Embedding 开关。
- **Behavior**：

- **Edge Case**：

- **Acceptance**：

## 10. 验收矩阵

### AI 基础设施

- 正常：Provider 切换、Prompt 模板加载、Job 执行
- 异常：Key 无效提示、Provider 超时重试、Job 失败重试
- 持久化：调用日志可查、成本可追踪

### AI 初始化

- 正常：两阶段分析、进度展示、结果落库
- 边界：无大纲、无正文、重新分析单章
- 异常：分析中断后恢复、跳过失败章节
- 恢复：已完成的阶段不重复执行

### 续写与写作

- 正常：单章续写、多章续写、选区改写/润色
- 边界：续写中断后继续、长篇超 token
- 异常：候选稿丢失恢复、连续修订失败
- 恢复：候选稿合并后进入正常保存链路

### 智能体工作流

- 正常：完整 5 Agent 流程
- 边界：某 Agent 调用失败、部分结果可用
- 异常：审稿不通过次数超限
- 恢复：跳过损坏的 Agent 步骤

### 审稿与修订

- 正常：7 维度审稿、自动修订
- 边界：无需修订的候选稿、修订耗尽快照
- 异常：审稿超时降级

### 剧情轨道与记忆

- 正常：四层轨道构建、Story Memory 更新
- 边界：不分卷的省略
- 持久化：刷新后轨道和记忆保持
- 恢复：增量更新不丢失历史数据

### AI 建议区

- 正常：建议分类展示、采纳/编辑/忽略
- 边界：建议过期、重复建议
- 冲突：Conflict Guard 对比界面

### @引用系统

- 正常：@联想/高亮/悬停、自动提取
- 边界：资产删除后引用展示
- 持久化：刷新后引用关联保持

## 11. 与 V1.1 的衔接约束

- V2.0 不改变 V1.1 的任何数据表结构
- V1.1 的 Local-First 保存链路不受 AI 功能影响
- 结构化资产（作品大纲 / 人物卡 / 时间线 / 伏笔）的写入必须经过用户确认，AI 建议不直接写入
- V1.1 的手动保存模式在 V2.0 中不变，AI 建议保存使用独立的异步写入流程
- Workbench 与 Legacy 域隔离继续有效

## 12. 默认决策

1. Kimi 作为默认的分析/规划/审稿模型，DeepSeek 作为默认的写作/修订模型
2. Story Memory 分层摘要的 token 上限：全书摘要 500t、卷摘要 800t、阶段摘要 1000t、章节摘要 200t
3. RAG 召回 Top-K 默认为 5
4. Context Pack 总 token 上限为模型上限的 80%
5. 候选稿自动丢弃时间：24 小时
6. AI 建议过期时间：7 天
7. 修订最大轮次：3 轮
8. 自动续写停止条件中，连续修订失败阈值默认为 3 次
9. 章节后处理管道串行执行，不支持并发
10. @引用联想限制：最多展示 10 条匹配结果
