## 1. 设计目标

### 1.1 核心定位

InkTrace V2.0 的定位是 **AI 辅助创作工具，不是 AI 自动创作工具**。

在三个参考项目的谱系中，InkTrace V2 站的位置：

```
纯手工                    AI 辅助（我们的位置）               AI 强驱动                  AI 全自动
  │                          │                                │                          │
novelWriter              InkTrace V2                      PlotPilot                  InkOS
(无 AI)                  (AI 建议，人确认)                  (AI 驱动，人审核)           (AI 全自动)
```

### 1.2 设计原则

| 原则 | 说明 |
|---|---|
| **AI 建议，人确认** | AI 生成的所有内容都是「建议」，需要用户显式确认后才落库 |
| **不改变正文输入流** | AI 交互不得干扰正在进行的写作输入 |
| **Asset 手动保存不变** | 即使 AI 填充了资产建议，用户仍需点击保存才持久化 |
| **Local-First 不变** | AI 功能不改变现有保存链路，离线不可用 AI 但现有写作不受影响 |
| **渐进增强** | 基础设施先搭，能力逐个挂载，不搞大爆炸式重构 |

### 1.3 参考项目的关键吸收

| 项目 | 吸收什么 | 拒绝什么 |
|---|---|---|
| **PlotPilot** | 后处理管道的设计理念、风格一致性检测、伏笔追踪 | AI 自动生成章节、知识图谱的复杂度 |
| **InkOS** | 审计维度的精简化设计理念、AI 痕迹检测 | 全自动 10-Agent 流水线、无人化写作 |
| **novelWriter** | `@` 标签引用系统、项目树的多文档组织 | 纯手工无 AI 的限制 |

```

│              AI Infrastructure (新增层)                │
│  ┌──────────┬──────────┬───────────┬──────────────┐  │
│  │ Provider │  Prompt  │ Streaming │ Token/Cost   │  │
│  │ 抽象层   │  模板管理 │  处理     │  追踪        │  │
│  └──────────┴──────────┴───────────┴──────────────┘  │
├──────────────────────────────────────────────────────┤
│                 Domain (Entities)                     │
│                  不变，V1.1 资产复用                   │

```

## 3. AI 基础设施层设计

### 3.1 为什么先搭基础设施

避免 V2.0 出现以下问题：

- 每个 AI 能力各自对接 LLM，重复代码
- Provider 切换需要改每个调用点
- Prompt 散落在代码中无法维护
- Token 消耗不可追踪

### 3.2 基础设施组件

#### 3.2.1 Provider 抽象层

```text
┌─────────────────────────────────────┐
│           AIProvider (抽象接口)       │
├─────────────────────────────────────┤
│  + chat(messages, options) → Stream │
│  + count_tokens(text) → int        │
│  + get_model_info() → ModelInfo     │
├─────────────────────────────────────┤
│          implements ▲  ▲            │
│                     │  │            │
│  ┌────────────┐ ┌────────────┐      │
│  │ KimiProvider│ │DeepSeekProv│      │
│  │            │ │ider        │      │
│  └────────────┘ └────────────┘      │
└─────────────────────────────────────┘
```

- 后端抽象接口，支持 Kimi / DeepSeek / Claude 等多种 Provider
- 每个 Provider 实现 token 计价、流式输出、错误重试
- 按能力配置不同模型（续写用 DeepSeek，分析用 Kimi）

#### 3.2.2 Prompt 模板管理

```text
application/ai/prompts/
├── templates/
│   ├── continue_writing.yaml      # AI 续写
│   ├── expand_paragraph.yaml      # AI 扩写
│   ├── rewrite_paragraph.yaml     # AI 重写
│   ├── summarize_paragraph.yaml   # AI 缩写
│   ├── outline_polish.yaml        # 大纲润色
│   ├── outline_expand.yaml        # 大纲扩写
│   ├── chapter_outline_gen.yaml   # 章节细纲生成
│   ├── character_extract.yaml     # 人物提取
│   ├── timeline_extract.yaml      # 时间线提取
│   ├── style_analysis.yaml        # 风格分析
│   ├── consistency_check.yaml     # 一致性检查
│   └── ai_trace_detect.yaml       # AI 痕迹检测
├── builder.py                     # Prompt 构建器
└── variables.py                   # 上下文变量注入
```

- Prompt 模板与代码分离，YAML 文件管理
- 上下文构建器自动注入：当前章节内容 + 大纲 + 人物卡 + 已写章节摘要
- 支持版本管理，Prompt 变更不涉及代码发布

#### 3.2.3 Streaming 处理

```text
用户触发 AI 能力
       │
       ▼
后端调用 Provider.chat(stream=True)
       │
       ▼
SSE EventSource 推送 tokens
       │
       ▼
前端流式写入编辑器（或预览区域）
       │
       ▼
用户确认 → 保留内容
用户拒绝 → 丢弃
```

- 后端统一 SSE 格式输出
- 前端统一流式处理组件
- 支持中断（用户按 Esc 停止生成）
- 支持逐段确认（生成长内容时分段展示，逐段同意/拒绝）

#### 3.2.4 Token 与成本追踪

```text
数据表：ai_usage_logs
├── id
├── capability         # 能力标识 (continue_writing / outline_polish / ...)
├── provider           # kimi / deepseek
├── model              # 具体模型名
├── prompt_tokens
├── completion_tokens
├── total_cost         # 估算成本
├── user_accepted      # 用户是否接受结果
└── created_at
```

- 每条 AI 调用记录日志
- 前端可展示今日/本周 AI 用量
- 可选设置月度 Token 预算上限

## 4. P0 能力：核心写作 AI

### 4.1 AI 续写

**触发位置**：正文 textarea 底部「AI 续写」按钮，或 `Ctrl+.` 快捷键

```text
┌──────────────────────────────────────────┐
│                                           │
│  主角推开了那扇门，眼前是一片……        │
│                                           │
│                                           │
│                                           │
│  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─  │
│  [AI 续写 Ctrl+.]  字数: 1,234 / 章      │
└──────────────────────────────────────────┘
```

**交互流程**：

1. 用户点击「AI 续写」
2. 前端构建上下文（当前章节全部内容 + 最近 X 章摘要 + 作品大纲）
3. 后端流式生成下文
4. 生成内容以**灰色占位文字**直接显示在 textarea 中光标位置
5. 用户按 `Tab` 接受整段、按 `Esc` 丢弃、继续打字则自动替换占位

**技术要点**：

- 流式写入 textarea 时禁止触发自动保存（AI 内容不是最终内容）
- 接受后才进入正常的 Local-First 保存链路
- Context 窗口需要控制，超过上限时使用章节摘要压缩

### 4.2 AI 扩写/重写/缩写

**触发位置**：选中正文文本后出现的浮动工具栏

```text
选中文本：「主角推开了那扇门」
                              ┌──────────────┐
                              │ 扩写  重写  缩写 │
                              └──────────────┘
```

**交互流程**：

1. 用户在 textarea 中选中一段文本
2. 浮动工具栏出现（扩写 / 重写 / 缩写 三个按钮）
3. 点击后，AI 生成替换建议
4. 建议在**预览浮窗**中展示（不直接替换正文）
5. 用户点击「应用」→ 替换原文 + 进入保存链路
6. 用户点击「跳过」→ 关闭预览

**三种模式的区别**：

| 模式 | 输入 | 输出 | 典型场景 |
|---|---|---|---|
| 扩写 | 短段落 | 更长的详细段落 | 描写不够细致 |
| 重写 | 段落 | 同长度的改写版 | 表达不满意 |
| 缩写 | 长段落 | 精简段落 | 节奏拖沓 |

### 4.3 AI 基础设施：Context 构建

续写、扩写、重写共享同一套 Context 构建规则：

```text
Context = {
    "当前章节": 完整全文,
    "作品大纲": work_outline.content_text (前 2000 字),
    "已写章节摘要": [ch1_summary, ch2_summary, ...],
    "主要人物": [name + 人物卡 description],
    "当前章节目录": title,
    "写作风格指导": (可选，来自风格分析),
}
```

- 后端 `ContextBuilder` 负责按 token 预算自动裁切
- 优先级：当前章节 > 最近章节摘要 > 人物卡 > 大纲
- 超出上限时，优先压缩章节摘要而非丢掉

## 5. P1 能力：结构化资产智能

### 5.1 章节后处理管道

**这是 V2.0 资产侧最重要的能力**。灵感来自 PlotPilot 的后处理管道设计。

**触发时机**：用户保存章节后（收到 200 OK），自动触发（或手动触发）

```text
用户保存章节 (PUT /api/v1/chapters/{id})
        │
        ▼
后端返回 200 OK
        │
        ▼
异步触发后处理管道（不影响用户继续写作）
        │
        ▼
   ┌─────────────┐
   │ 章节摘要生成  │──→ 存入章节元数据
   ├─────────────┤
   │ 人物提及提取  │──→ 更新人物卡引用
   ├─────────────┤
   │ 时间事件提取  │──→ 生成时间线建议
   ├─────────────┤
   │ 伏笔线索检测  │──→ 标记潜在伏笔
   └─────────────┘
        │
        ▼
结果写入 AI 建议区（Asset Drawer 内「AI 建议」Tab）
用户审阅后决定是否采纳
```

**关键约束**：

- **异步执行**：不阻塞正文保存响应
- **仅写建议区**：不直接写资产表，不触发 dirty 状态
- **用户确认后才落库**：采纳后写入资产表，进入正常 dirty → save 链路

### 5.2 大纲润色与扩写

**触发位置**：OutlinePanel 内的「AI 润色」和「AI 扩写」按钮

```text
┌──────────────────────────────────┐
│ 作品大纲              [导入] [AI] │  ← AI 下拉菜单
├──────────────────────────────────┤
│                                  │
│        作品大纲 textarea          │
│                                  │
├──────────────────────────────────┤
│ 当前字符数：12,345               │
└──────────────────────────────────┘

点击 [AI] → 展开菜单：
  ├── AI 润色
  ├── AI 扩写
  └── AI 生成章节细纲建议
```

**交互流程**：

1. 用户点击「AI 润色」或「AI 扩写」
2. 当前大纲内容发送到后端
3. AI 生成的结果在 outline textarea 中以**灰色对比文本**展示
4. 用户按 `Tab` 接受、按 `Esc` 拒绝
5. 接受后进入 dirty 状态，用户仍需点击「保存作品大纲」持久化

### 5.3 章节细纲生成建议

**触发位置**：章节大纲区域的「AI 生成建议」按钮

**输入**：

- 作品大纲 (`work_outline.content_text`)
- 当前章节之前的已写内容（摘要级别）
- 当前章节标题

**输出**：

- 当前章节的细纲建议（3-5 个要点）
- 展示在章节大纲 textarea 上方的建议区
- 用户可逐条采纳/拒绝/修改

### 5.4 风格一致性检测

**触发位置**：Asset Drawer 内独立面板「写作分析」

```text
写作分析面板
├── 风格一致性（风险项列表）
│   ├── ⚠️ 第 3 章与第 12 章的人物语气不一致
│   ├── ⚠️ 第 8 章描写密度明显低于全书平均
│   └── ⚠️ "青云山"在第 7 章后改为"青山"（可能是笔误）
├── 节奏分析
│   ├── 平均段落长度：85 字
│   ├── 对话占比：32%
│   └── 章节字数分布折线图
└── 高频词统计
    ├── Top10 实词列表
    └── 建议替换的过度使用词
```

**数据来源**：

- 全文统计（传统 NLP，不需要 LLM）
- 风格对比（LLM 分析，按需调用）
- 命名一致性检查（正则匹配 + 人物卡对照）

## 6. P2 能力：@标签引用系统

### 6.1 设计目标

打通正文与结构化资产的连接。灵感来自 novelWriter 的 `@` 标签系统。

### 6.2 正文中的 `@` 引用

```text
主角走进了 @青云山，这里是 @青云宗 的山门所在。
@张峰 已经在门口等候多时。
```

- 输入 `@` 时弹出联想菜单：匹配人物/地点/物品名称
- 选中的 `@实体` 在正文中高亮显示
- 鼠标悬停时弹出浮窗显示实体摘要（从 Asset 读取）

### 6.3 后端支持

```text
GET /api/v2/works/{work_id}/mentions?q=青云
→ 返回匹配的人物/地点/事件列表

POST /api/v2/works/{work_id}/chapters/{chapter_id}/mentions
→ 保存当前章节中所有 @ 引用关系
→ 存入新表 chapter_mentions
```

新增数据表：

```text
chapter_mentions
├── id
├── chapter_id
├── asset_type        # character / location / event
├── asset_id          # 对应资产的 ID
├── mention_text      # 被引用的文本（如"青云宗"）
├── offset_start      # 在章节中的起始位置
├── offset_end        # 结束位置
└── created_at
```

### 6.4 前端交互

| 操作 | 行为 |
|---|---|
| 输入 `@` | 弹出联想下拉框，搜索人物/地点/物品 |
| 选择实体 | 插入 `@实体名`，在文本中高亮 |
| 悬停 `@实体` | 弹出浮窗显示实体卡片摘要 |
| 保存章节 | 自动提取所有 `@` 引用并调用 mentions API |
| 删除章节 | 级联清理 mentions |

### 6.5 @引用与 AI 的联动

后处理管道在提取人物和事件时，自动构建 `@` 引用关系：

```
AI 提取：「本章提到张峰、青云山」
         │
         ▼
自动在章节中建立张峰 → character.id 的引用
自动在章节中建立青云山 → (新增地点 asset 建议)
自动在 timeline 中标记时间事件
```

## 7. P3 能力：分析看板

### 7.1 入口位置

Asset Drawer 新增 Tab：「分析」

### 7.2 面板内容

#### 7.2.1 写作统计（前端本地计算，无需 AI）

- 今日新增字数
- 本周写作趋势（折线图）
- 总字数 / 章节数 / 平均每章字数
- 最长章节 / 最短章节

#### 7.2.2 节奏分析（传统 NLP，无需 LLM）

- 全章节字数条形图
- 对话段落占比
- 段落长度分布
- 标点符号使用统计

#### 7.2.3 一致性检查（需要 LLM，按需触发）

用户点击「开始检查」：

- 人物名前后一致性（正则匹配人物卡即可，不需要 LLM）
- 关键物品前后名称一致
- 时间线逻辑冲突（如角色在第 3 章已死，第 8 章又出现）
- 伏笔未回收提醒

#### 7.2.4 AI 痕迹检测（需要 LLM，按需触发）

参考 InkOS 的类似功能，检测：

- 高频 LLM 句式（"然而"、"值得注意的是"、"无言中"）
- 段落结构过于均匀（每段 3-4 句）
- 缺乏具象细节的描述

## 8. 交互模式汇总

### 8.1 三种 AI 交互模式

| 模式 | 适用场景 | 用户体验 | 对写作流的侵入 |
|---|---|---|---|
| **Inline 流式** | 续写、润色 | AI 内容直接在 textarea 展示，Tab 接受 | 低（在当前编辑区） |
| **浮动工具栏** | 扩写/重写/缩写 | 选中文本 → 工具栏 → 预览浮窗 | 低（在当前编辑区） |
| **面板建议区** | 资产提取、分析 | 结果在 Drawer 内展示，用户审阅 | 极低（在侧面板） |

### 8.2 AI 与保存链路的关系

```
Inline 流式生成
    │ 用户按 Tab 接受
    ▼
写入 textarea（进入正常输入流）
    │
    ▼
触发本地草稿保存（现有 Local-First 链路不变）
    │
    ▼
防抖后调 API 保存
    │
    ▼
200 OK → 可选触发后处理管道（异步）
```

## 9. 路由与 API 设计

### 9.1 API 前缀

V2.0 AI API 统一使用 `/api/v2/ai/*` 前缀，与 V1.1 的 `/api/v1/*` 隔离。

### 9.2 API 清单

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/v2/ai/continue-writing` | AI 续写（流式 SSE） |
| POST | `/api/v2/ai/expand` | AI 扩写（流式 SSE） |
| POST | `/api/v2/ai/rewrite` | AI 重写（流式 SSE） |
| POST | `/api/v2/ai/summarize` | AI 缩写（流式 SSE） |
| POST | `/api/v2/ai/outline-polish` | 大纲润色（流式 SSE） |

## 11. 明确禁止的行为

### 11.1 V2.0 不做的

- ❌ AI 自动创建章节（大纲→章节拆分可以建议，但不自动创建）
- ❌ AI 全自动写作（InkOS 模式）
- ❌ AI 自动替换正文（所有生成内容需要用户显式确认）
- ❌ AI 自动覆盖资产（后处理管道只写建议区）
- ❌ 改变 Local-First 保存链路
- ❌ 改变手动保存的资产模式
- ❌ 引入向量数据库 / 知识图谱（PlotPilot 的复杂度，V2.0 不需要）

### 11.2 V2.0 保持 V1.1 的边界

- Workbench 与 Legacy 域隔离继续有效
- V1.1 的非 AI 功能完全不受影响
- 离线时 AI 能力不可用，但写作不受影响
- 结构化资产仍然是手动保存，AI 只填充建议
