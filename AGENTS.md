# 规则.md（InkTrace 项目专用实现约束）

> 目的：防止实现阶段脱离设计文档“主观发挥”。
> 生效范围：本仓库所有后续实现任务（后端、前端、测试、脚本）。
> 版本：v1.1  
> 生效日期：2026-05-15  
> 更新说明：增强实现前约束维度、测试执行规则、停止确认机制，减少“架构约束/安全红线”语义重复。

---

## 0. 总原则（必须遵守）

1. **只按已冻结文档实现，不按个人想象实现。**
2. **先对齐文档再写代码。**未完成“实现前确认”不得改代码。
3. **超出当前阶段范围的需求一律拒绝实现**，只登记“待确认点/后续阶段”。
4. **冲突不自行拍板**：文档冲突时立即停止并上报。
5. **安全边界优先于功能完整性**：任何会破坏 user_action、门控、权限、审计边界的改动一律禁止。

---

## 1. 文档优先级与裁决顺序

当文档、代码、口头需求不一致时，按以下顺序裁决：

1. `docs/01_requirements/InkTrace-V2.0-需求规格说明书.md`
2. `docs/02_architecture/InkTrace-V2.0-架构设计说明书.md`
3. `docs/07_overview/InkTrace-V2.0-概要设计说明书.md`
4. 当前阶段详细设计（P0/P1 对应冻结文档）
5. 当前阶段开发计划（如有）
6. 现有代码

**结论**：代码必须向文档对齐，不允许“代码现状反向定义设计”。

## 3. 实现前强制流程（不允许跳过）

每次实现任务开始前，必须输出并完成以下检查：

### 3.1 实现前确认（必填）

- 本次任务目标（1-3 句）
- 读取文档清单（精确到路径）
- 允许修改范围（目录/文件类型）
- 禁止修改范围（目录/文件类型）
- 关键约束摘录（必须覆盖以下维度，来自文档原意）：
  - 门控边界（如 DirectionSelection / PlanConfirmation / HumanReviewGate / MemoryReviewGate）
  - 状态机约束（如 ready/degraded/blocked、pending/confirmed/stale）
  - 权限约束（如 caller_type、user_action、Tool/side_effect_level）
  - 数据边界（是否允许写正式正文、是否只允许 CandidateDraft）
  - 安全日志约束（不得记录完整 Prompt/API Key/完整 ContextPack）
- 冲突/缺口（有则停止）

### 3.2 进入编码门槛
若任一条件不满足，**禁止写代码**：
- 未读取对应阶段详细设计
- 不清楚 caller_type / user_action 边界
- 不清楚 blocked/degraded 规则
- 不清楚是否会影响正式正文或正式资产

---

## 4. 实现范围控制规则

1. 只改“任务明确授权”范围内文件。
2. 禁止顺手重构、禁止顺手改风格、禁止顺手改无关模块。
3. 禁止提前实现后续阶段能力（P1/P2 越界）。
4. 禁止以“未来可能需要”为理由增加接口/字段/状态机。
5. 无文档定义的实体、状态、错误码，不得私自新增到正式链路。

---

## 5. 架构与分层硬约束

> 说明：本章定义“分层与调用路径”底线；安全结果型红线见第 6 章。两章互为补充，任一违反即阻断。

1. 不允许跨层旁路调用（UI/API 直连底层存储/Provider）。
2. 不允许绕过受控门面（如 ToolFacade、Application Service 边界）。
3. 不允许业务逻辑散落到 Presentation 层。
4. Domain 不依赖 Infrastructure。
5. 不允许直接使用 Provider SDK 绕过 ModelRouter/受控路径。

---

## 6. 安全红线（违反即阻断）

以下任一项触发即判定为阻断问题：

1. AI 自动写正式正文。
2. AI 自动 accept/reject/apply CandidateDraft。
3. workflow/agent/system 伪造 user_action。
4. 绕过 HumanReviewGate / MemoryReviewGate / ConflictGuard。
5. 暴露 API Key 明文。
6. 日志记录完整 Prompt / 完整 ContextPack / 完整正文 / 完整候选稿。
7. blocked 状态被伪装成 ready。

---

## 7. 状态机与错误码约束

1. 状态机必须与对应详细设计一致，不得自创“近似状态”。
2. 错误码只可在设计定义集合内扩展（且需在文档登记）。
3. warning 与 error 不得混用。
4. 任何 `partial_success` 必须满足“存在可交付 result_ref”的前提。

---

## 8. 测试约束（实现即补测）

每次实现必须覆盖：
1. 正常路径
2. 错误路径
3. 边界路径
4. 权限/门控路径（caller_type、user_action）
5. 安全红线路径（不得越权、不得写正式资产）

执行规则：
1. 测试文件应落在项目约定测试目录（如 `tests/` 及其子目录），并与改动模块同域放置。
2. 实现者（含 AI coding agent）必须在本地执行与本任务相关的测试命令。
3. 若项目有 CI，默认“CI 关键检查通过”作为可合并前置条件；如当前任务环境无 CI 回执，必须在报告中明确“待 CI 复验”。
4. 若无法运行测试（环境缺依赖/权限限制/外部服务不可用），必须：
   - 明确无法执行的原因；
   - 提供可复现的测试命令；
   - 标记风险级别与影响范围；
   - 给出“需人工补跑测试”清单；
   - 在补跑前不得宣称“已完全验收通过”。

并在完成报告中提供：
- 测试命令
- 结果摘要
- 未覆盖缺口

---

## 9. 交付格式约束（实现完成后）

必须输出“实现完成报告”：

1. 修改文件清单（文件 + 改动目的）
2. 文档对齐清单（改动点对应哪条设计）
3. 测试结果（命令 + 通过/失败）
4. 未完成项与风险
5. 是否可进入下一阶段（可以/不建议 + 原因）

---

## 10. 冲突与缺失处理（停止规则）

遇到以下情况必须停止编码并先确认：

1. 关键文档缺失或版本不明。
2. 同一能力在多份文档中语义冲突。
3. 当前任务请求与阶段边界冲突。
4. 代码现状与文档严重背离且无法安全渐进修复。

确认对象：
- 默认确认对象为：任务提出者 / 人工审核者（人类负责人）。
- 停止后必须等待人工回复，不得自行推断继续。

输出格式：
- 冲突点
- 影响范围
- 建议裁决选项（不直接实施）

## 11. 本项目专项补充（InkTrace）

1. `docs/03_design/V2/InkTrace-V2.0-P0-*` 是 P0 正式依据路径，不得误判为错误路径。
2. P0 与 P1 并行存在时：
   - P0 主链路保持可用与兼容；
   - P1 新能力不得破坏 P0 安全边界。
3. DirectionSelection / PlanConfirmation / HumanReviewGate / MemoryReviewGate 是不同确认门，禁止混用语义。
4. CandidateDraft 必须保持隔离层属性，未 apply 不得进入正式正文链路。
5. UI 实现必须同时遵守：
   - `InkTrace-V2.0-P1-UI-界面与交互设计.md`
   - `InkTrace-DESIGN.md`
   - 对应阶段 API 集成边界文档
